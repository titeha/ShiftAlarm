# Синхронизация экрана остановки с сервисом будильника

## Назначение документа

Этот документ фиксирует правило закрытия полноэкранного окна сигнала.

У пользователя есть несколько способов остановить будильник:

```text
кнопка «Стоп» на полноэкранном экране;
кнопка «Стоп» в уведомлении;
смахивание уведомления, если прошивка это допускает;
остановка сервиса системой.
```

Все представления одного сигнала должны исчезать согласованно.

## Проблема

Кнопка уведомления отправляла `ACTION_STOP` напрямую в `AlarmService`.

Сервис прекращал звук, вибрацию и удалял foreground-уведомление, но
`AlarmActivity` об этом не знала. Поэтому полноэкранное окно оставалось
открытым до отдельного нажатия «Стоп».

## Принятое правило

Источник истины об активном сигнале — `AlarmService`.

Сервис публикует состояние через:

```text
AlarmSignalState.isRinging
```

Переходы:

```text
обычный запуск сервиса -> true;
ACTION_STOP -> false;
onDestroy сервиса -> false.
```

`AlarmActivity` наблюдает состояние. Когда оно становится `false`, Activity
вызывает `finish()`.

## Почему используется StateFlow

Состояние действует только внутри процесса и не требует системного broadcast.

Преимущества:

```text
экран сразу получает текущее значение;
не нужно регистрировать и снимать BroadcastReceiver;
внутреннее событие не экспортируется наружу;
логику переходов можно проверить unit-тестом;
новая Activity, открытая после остановки сервиса, сразу закроется.
```

## Почему состояние не сохраняется

Активный звонок — краткоживущее состояние процесса.

Если процесс уничтожен:

```text
AlarmService больше не играет;
AlarmActivity также уничтожена;
старое состояние восстанавливать нельзя.
```

Поэтому DataStore, Room и SharedPreferences здесь не нужны.

## Проверка тестами

Unit-тесты проверяют:

```text
начальное неактивное состояние;
переход в active при запуске;
переход в inactive при остановке.
```

Главный пользовательский сценарий проверяется на устройстве, потому что он
включает foreground service, уведомление и Activity.

## Правило для будущих изменений

Новый способ остановки сигнала должен останавливать `AlarmService`.

Он не должен отдельно управлять каждым UI-представлением. Сервис меняет
единое состояние, а экран и уведомление реагируют на него.
