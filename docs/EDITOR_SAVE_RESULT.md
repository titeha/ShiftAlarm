# Результат сохранения редактора будильника

## Назначение документа

Этот документ фиксирует поведение редактора во время сохранения и при технической ошибке.

Пользователь не должен видеть закрытый редактор до того, как приложение узнало результат записи данных. Повторное нажатие кнопки не должно запускать несколько параллельных транзакций.

## Этапы операции

Сохранение состоит из двух этапов:

```text
1. Транзакционная запись AlarmEntity, periods и overrides в Room.
2. Перепланирование системного сигнала AlarmManager.
```

Эти этапы различаются, потому что Room и AlarmManager не образуют одну общую транзакцию.

## Во время сохранения

После нажатия «Сохранить»:

```text
кнопка показывает состояние «Сохранение»;
повторное нажатие недоступно;
кнопка «Отмена» недоступна;
системная кнопка «Назад» не закрывает редактор;
черновик остаётся на экране до результата.
```

## Успешная запись и перепланирование

Если оба этапа завершились:

```text
редактор закрывается;
пользователь возвращается в список;
результат сохранения очищается перед следующей сессией.
```

## Ошибка записи данных

Если транзакция Room не завершилась:

```text
редактор остаётся открыт;
кнопки снова становятся доступны;
показывается сообщение, что изменения не применены;
пользователь может исправить данные или повторить сохранение.
```

Технический текст исключения не показывается пользователю. Он записывается в Logcat.

## Данные сохранены, но сигнал не перепланирован

Если Room завершил транзакцию, а AlarmManager отказал:

```text
данные уже сохранены;
редактор закрывается;
в списке показывается заметное предупреждение;
пользователю предлагается проверить разрешения и повторно сохранить запись.
```

Такой результат нельзя выдавать за полную ошибку сохранения: молчаливый откат БД потребовал бы восстановить старый будильник, периоды, правки и прежний системный сигнал. Ошибка отката могла бы повредить данные сильнее.

## Состояния

`AlarmListViewModel` публикует:

```text
Idle
Saving
Saved
Failed
```

`Saved` может содержать предупреждение о системном сигнале.

## Чистая проверяемая часть

Последовательность двух этапов вынесена в:

```text
AlarmSaveCoordinator
```

Unit-тесты проверяют:

```text
успешную запись и передачу нового id планировщику;
запрет перепланирования после ошибки БД;
сохранённые данные с предупреждением после ошибки AlarmManager;
правильный этап технической ошибки;
проброс CancellationException без превращения в пользовательскую ошибку.
```

## Ограничения текущего шага

Этот шаг не сохраняет черновик после уничтожения процесса.

Сохранение состояния формы через ViewModel и `SavedStateHandle` выполняется отдельно.

## Правило для будущих изменений

Новая операция, выполняемая после транзакции БД, не должна заставлять UI ложно сообщать, что данные не сохранены.

Интерфейс обязан различать:

```text
данные не записаны;
данные записаны, но вторичная операция не выполнена.
```
