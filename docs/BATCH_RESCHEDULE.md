# Изоляция ошибок массового перепланирования

## Назначение документа

Этот документ фиксирует поведение ShiftAlarm при массовом восстановлении
системных сигналов.

Массовое перепланирование выполняется:

```text
после перезагрузки устройства;
после обновления приложения;
после изменения времени;
после изменения часового пояса;
после обновления производственного календаря.
```

## Проблема

Ранний вариант выполнял обычный последовательный цикл:

```kotlin
alarms.forEach { alarm ->
    reschedule(context, repo, alarm)
}
```

Если один будильник выбрасывал исключение, корутина завершалась и все записи
после него оставались без повторной постановки в `AlarmManager`.

Для будильника это опасная зависимость:

```text
ошибка одной повреждённой записи
не должна лишать системного сигнала остальные корректные записи.
```

## Принятое правило

Каждый будильник обрабатывается независимо.

Результат одного элемента:

```text
успешно — увеличивается число успешных операций;
ошибка — сохраняются alarmId и короткая причина;
отмена корутины — весь пакет прекращается корректно.
```

После ошибки обычного элемента цикл продолжается.

## Отчёт

`AlarmRescheduleReport` содержит:

```text
общее число будильников;
число успешных операций;
список ошибок с alarmId;
вычисляемое число ошибок;
признак полного успеха.
```

Отчёт используется:

```text
для Logcat;
для диагностического журнала;
для сводки BootReceiver;
для будущего экрана готовности.
```

## Диагностика

`AlarmScheduler.rescheduleAll` записывает отдельное событие `ERROR`
для каждого неудачного alarmId.

`BootReceiver` записывает итог:

```text
причина системного события;
сколько будильников успешно;
сколько завершилось ошибкой;
несколько первых ошибочных id.
```

Ошибка чтения всего списка из Room по-прежнему обрабатывается внешним
`try/catch` receiver'а, потому что в таком случае набор элементов ещё неизвестен.

## Отмена корутины

`CancellationException` не записывается как ошибка конкретного будильника.

Отмена означает, что владелец корутины прекратил операцию. Она пробрасывается
наружу, чтобы не продолжать работу после завершения жизненного цикла.

## Проверка тестами

Unit-тесты проверяют:

```text
пустой набор;
полностью успешный пакет;
сохранение порядка;
ошибку среднего элемента;
продолжение обработки после ошибки;
несколько ошибок;
короткий список alarmId;
проброс CancellationException.
```

## Проверка на устройстве

Искусственно повреждать будильник не требуется.

Достаточно smoke-проверки:

```text
создать два включённых будильника;
установить новую debug-сборку поверх старой;
открыть диагностику;
найти событие системного перепланирования;
убедиться, что в сводке нет ошибок;
убедиться, что оба ближайших сигнала отображаются корректно.
```

Ошибка одного элемента покрывается unit-тестом чистого пакетного исполнителя.

## Правило для будущих изменений

Любая массовая операция над независимыми будильниками должна явно решить:

```text
должна ли ошибка одного элемента остановить остальные;
как выглядит отчёт;
куда записывается диагностика;
как обрабатывается CancellationException.
```
