# ShiftAlarm: дорожная карта к состоянию «10/10»

## Назначение документа

Этот документ фиксирует, что нужно сделать, чтобы ShiftAlarm из хорошего предрелизного проекта превратился в устойчивый production-уровень.

Под «10/10» здесь не понимается абсолютная гарантия срабатывания на любом Android-устройстве при любых ограничениях ОС и прошивки. Для Android-будильника это недостижимая формулировка: часть поведения зависит от версии ОС, разрешений, режима энергосбережения, OEM-прошивки, уведомлений, full-screen intent и пользовательских настроек.

Под «10/10» в этом документе понимается другое: проект доведён до состояния, где архитектура, движок, данные, тесты, документация, Android-интеграция, диагностика и релизный процесс соответствуют хорошему production-уровню для выбранной ниши.

Выбранная ниша: будильник для рабочих, учащихся, сменщиков и вахтовиков, где основная ценность — корректная работа со сменными графиками, ночными сменами, отгулами, отпусками, больничными, праздниками и переносами.

## Текущее состояние

Проект уже прошёл несколько важных стабилизационных этапов.

Закрыты или частично закрыты следующие риски:

```text
- исправлена привязка звонков в кастомных ночных сменах;
- добавлена документация по ночным сменам;
- исправлено сохранение будильников без удаления связанных periods/overrides;
- AlarmReceiver больше не запускает звонок до проверки актуальности срабатывания;
- добавлена валидация внешнего ACTION_SET_ALARM;
- добавлено перепланирование после системных событий;
- добавлена вибрация как запасной канал сигнала;
- добавлено защитное чтение сохранённых расписаний и правок календаря;
- опасные места начали обрастать unit/instrumentation-тестами и документацией.
```

Текущая субъективная оценка ядра проекта:

```text
движок расписаний:      8.5/10
сохранность данных:     8/10
Android-надёжность:     7/10
безопасность:           7.5/10
тесты:                  7/10
документация:           8/10
UI:                     5–6/10
общее состояние ядра:   7.5–8/10
```

UI пока не является главным фокусом. Основной приоритет — движок, данные, планирование, надёжность срабатывания и проверяемость.

## Что означает «10/10» для этого проекта

Для ShiftAlarm «10/10» означает:

```text
- пользовательские расписания не теряются при обновлениях и редактировании;
- движок расписаний предсказуемо работает на сменах, ночах, отгулах, отпусках и праздниках;
- срабатывание будильника защищено от устаревших PendingIntent;
- после перезагрузки, обновления приложения, смены времени и часового пояса будильники перепланируются;
- приложение явно показывает пользователю проблемы с разрешениями и системными ограничениями;
- критичная логика покрыта unit/instrumentation-тестами;
- есть матрица ручной проверки на реальном устройстве;
- есть CI, который автоматически ловит регрессии;
- есть понятная документация по архитектуре, данным, Android-интеграции и релизам;
- есть диагностика, которая помогает понять, почему будильник сработал или не сработал;
- релизный процесс воспроизводим и не зависит от памяти разработчика.
```

## Блок 1. CI и автоматические проверки

### Проблема

Локальные тесты полезны, но они зависят от конкретной машины. Для production-уровня проверки должны запускаться автоматически на каждый push и pull request.

### Целевое состояние

В репозитории есть GitHub Actions workflow, который выполняет:

```text
- ./gradlew testDebugUnitTest;
- ./gradlew assembleDebug;
- ./gradlew lintDebug;
- ./gradlew assembleDebugAndroidTest;
- проверку миграций БД, если она может выполняться в CI;
- публикацию отчётов тестов и lint-отчётов как artifacts.
```

### Критерий готовности

```text
- любой push в master проверяется автоматически;
- pull request нельзя безопасно сливать без зелёных тестов;
- отчёты доступны из GitHub Actions;
- локальные инструкции совпадают с CI-командами.
```

### Приоритет

Высокий. Это один из следующих шагов после текущей стабилизации ядра.

## Блок 2. Сверка manifest и системных событий

### Проблема

Для будильника важно, чтобы manifest реально содержал все системные события, которые должны приводить к перепланированию.

Особенно нужно проверить:

```text
android.intent.action.BOOT_COMPLETED
android.intent.action.MY_PACKAGE_REPLACED
android.intent.action.TIME_SET
android.intent.action.TIMEZONE_CHANGED
```

### Целевое состояние

`BootReceiver` подписан на нужные события, а чистая логика выбора событий покрыта unit-тестами.

### Критерий готовности

```text
- manifest содержит все поддерживаемые действия;
- SystemRescheduleActions покрыт тестами;
- документация SYSTEM_RESCHEDULE.md совпадает с manifest;
- ручная проверка перезагрузки устройства описана в матрице проверок.
```

### Приоритет

Высокий. Это нужно сверить до перехода к более крупным функциям.

## Блок 3. Разрешения и деградация поведения

### Проблема

Будильник зависит от системных возможностей Android:

```text
- уведомления;
- full-screen intent;
- точные будильники;
- foreground service;
- вибрация;
- звук;
- ограничения энергосбережения.
```

Если пользователь запретил нужные разрешения или прошивка ограничила приложение, это нужно явно показать.

### Целевое состояние

В приложении есть слой диагностики разрешений, который умеет определить:

```text
- разрешены ли уведомления;
- можно ли показывать full-screen intent;
- доступна ли точная постановка будильника;
- есть ли вибратор;
- доступен ли системный звук будильника;
- не требует ли устройство ручной настройки энергосбережения.
```

UI должен не просто падать или молча не звонить, а объяснять пользователю проблему и давать путь к системным настройкам.

### Критерий готовности

```text
- есть отдельный диагностический слой без смешивания с UI;
- критичные проверки покрыты тестируемой чистой логикой там, где это возможно;
- есть экран или блок состояния «Готовность будильника»;
- документация описывает, что значит каждый статус.
```

### Приоритет

Высокий, но после CI и сверки manifest.

## Блок 4. Матрица ручной проверки на устройстве

### Проблема

Будильник нельзя полностью проверить только unit-тестами. Нужны реальные проверки на телефоне.

### Целевое состояние

Есть документ:

```text
docs/DEVICE_TEST_MATRIX.md
```

Он содержит ручные сценарии:

```text
- обычный будильник на ближайшее время;
- будильник при заблокированном экране;
- будильник при разблокированном экране;
- остановка через экран;
- остановка через уведомление;
- звук + вибрация;
- изменение времени до срабатывания;
- удаление будильника до срабатывания;
- выключение будильника до срабатывания;
- перезагрузка устройства;
- обновление приложения поверх старой версии;
- ручное изменение системного времени;
- смена часового пояса;
- режим энергосбережения;
- выключенные уведомления;
- full-screen intent запрещён;
- ночные смены;
- отгулы/периоды без будильника;
- праздники, когда праздничный модуль будет подключён.
```

### Критерий готовности

```text
- матрица есть в docs;
- сценарии можно выполнить без знания внутреннего кода;
- результат каждого сценария однозначен: прошло / не прошло;
- перед релизом матрица реально прогоняется хотя бы на одном устройстве.
```

### Приоритет

Высокий. Особенно перед первым публичным APK.

## Блок 5. Движок расписаний и инварианты

### Проблема

Сейчас движок уже покрыт большим числом точечных тестов. Для production-уровня нужны ещё инвариантные проверки.

### Целевое состояние

Добавлены тесты, которые проверяют общие свойства, а не только отдельные примеры.

Инварианты:

```text
- AlarmTimes.next всегда возвращает время строго после from;
- выключенный будильник не планируется;
- разовый будильник после срабатывания корректно выключается или удаляется;
- период без будильника не создаёт звонки внутри периода;
- явное исключение имеет приоритет над периодом без будильника;
- swap/override имеет ожидаемый приоритет;
- ночная смена звонит накануне;
- следующая ночная смена не теряет звонок из-за отмены предыдущей;
- повреждённый cycleSpec не роняет планировщик;
- повреждённые overrides пропускаются;
- смена часового пояса не ломает локальную дату смены;
- большие диапазоны дат не приводят к бесконечным поискам.
```

### Критерий готовности

```text
- есть отдельный набор tests для инвариантов движка;
- тесты не зависят от Android runtime;
- сценарии читаемы и описаны в документации;
- при изменении движка тесты быстро показывают, что сломалось.
```

### Приоритет

Средне-высокий. Это желательно сделать до праздничного модуля, потому что праздники усложнят правила.

## Блок 6. Праздники и производственный календарь

### Проблема

README уже описывает учёт праздников и производственный календарь, но загрузка, парсинг и полноценное планирование праздников ещё не внедрены.

### Целевое состояние

Праздничный модуль разделён на слои:

```text
- чистая модель праздничного календаря;
- источник данных;
- парсер;
- локальный кеш;
- версия календаря;
- применение к расписанию;
- UI для включения и просмотра;
- тесты.
```

Главное правило: звонок не должен зависеть от сети в момент срабатывания. Сеть может обновлять справочник заранее, но не должна участвовать в ответе на вопрос «звонить сейчас или нет».

### Критерий готовности

```text
- есть локальный кеш праздничных данных;
- при отсутствии сети приложение продолжает работать по последнему известному календарю;
- формат данных валидируется;
- ошибки загрузки не ломают базовый будильник;
- есть тесты на праздничные дни, переносы и короткие недели;
- пользователь может понять, какой календарь применён.
```

### Приоритет

Средний. Праздники важны для продукта, но их лучше внедрять после стабилизации ядра, CI и диагностики.

## Блок 7. Backup/restore и приватность

### Проблема

Расписания смен, отпуска, больничные, отгулы и ручные правки являются личными данными. В manifest включён backup, поэтому нужно осознанно решить, какие данные можно переносить и как приложение ведёт себя после восстановления.

### Целевое состояние

Есть политика backup/restore:

```text
- какие таблицы попадают в backup;
- какие данные не должны попадать в backup;
- что происходит после restore на новом устройстве;
- как проверяются разрешения после restore;
- как перепланируются будильники после restore;
- что пользователь должен увидеть после восстановления.
```

### Критерий готовности

```text
- есть docs/BACKUP_AND_RESTORE.md;
- manifest и backup rules соответствуют документу;
- после restore приложение не считает, что системные разрешения автоматически сохранены;
- будильники перепланируются из базы;
- пользовательские данные не теряются.
```

### Приоритет

Средний. Перед публичной публикацией — обязательный.

## Блок 8. Диагностика внутри приложения

### Проблема

Если будильник не сработал, без диагностики трудно понять причину: ошибка расписания, разрешения, энергосбережение, звук, full-screen intent, сбой перепланирования или пользовательская настройка.

### Целевое состояние

Есть экран или служебный раздел «Диагностика».

Он показывает:

```text
- ближайшее рассчитанное срабатывание по движку;
- ближайшее поставленное системное срабатывание;
- статус уведомлений;
- статус full-screen intent;
- статус точных будильников;
- наличие вибрации;
- состояние системного звука будильника;
- последнюю причину перепланирования;
- время последнего успешного срабатывания;
- версию приложения;
- версию схемы БД;
- количество включённых будильников;
- ошибки последнего планирования, если были.
```

### Критерий готовности

```text
- диагностика помогает самому разработчику при тестировании;
- часть данных можно скопировать в текстовый отчёт;
- пользователь может понять, что мешает надёжной работе будильника;
- документация описывает значения диагностических статусов.
```

### Приоритет

Средний. Очень полезно до массового тестирования на устройствах.

## Блок 9. Релизный процесс

### Проблема

Даже хороший код можно сломать плохим релизным процессом.

### Целевое состояние

Есть документированный release checklist:

```text
- обновить versionCode/versionName;
- прогнать unit-тесты;
- прогнать instrumentation-тесты;
- прогнать lint;
- собрать release APK/AAB;
- проверить подпись;
- прогнать device matrix;
- обновить changelog;
- проверить privacy policy;
- проверить backup policy;
- создать tag;
- сохранить artifacts.
```

### Критерий готовности

```text
- релиз можно повторить по инструкции;
- versionCode увеличивается предсказуемо;
- changelog соответствует фактическим изменениям;
- release-сборка не зависит от локальных случайных файлов;
- секреты и ключи не попадают в репозиторий.
```

### Приоритет

Средний. Обязателен перед первой публичной публикацией.

## Блок 10. UI и UX

### Проблема

UI пока не главный фокус, но для production-уровня он должен защищать пользователя от непонятных и опасных настроек.

### Целевое состояние

UI должен ясно объяснять:

```text
- почему ночная смена звонит накануне;
- что означает точка будильника в календаре;
- какой звонок отменяет выходной/отгул;
- какие дни затрагивает период без будильника;
- почему будильник может не сработать без разрешений;
- когда праздник влияет на график;
- что будет после удаления будильника;
- что произойдёт с разовым будильником после срабатывания.
```

### Критерий готовности

```text
- пользователь может настроить сменный график без знания внутренней модели;
- календарь не вводит в заблуждение по ночным сменам;
- опасные действия имеют понятное подтверждение;
- ошибки ввода показываются рядом с причиной;
- UI не создаёт некорректные данные в БД.
```

### Приоритет

Средний-низкий сейчас, высокий перед публичным релизом.

## Рекомендуемый порядок работ

Ближайший порядок:

```text
1. Добавить CI на push/pull_request.
2. Сверить manifest и системные события перепланирования.
3. Добавить DEVICE_TEST_MATRIX.md.
4. Добавить диагностику разрешений и состояния будильника.
5. Расширить инвариантные тесты движка расписаний.
6. Описать backup/restore policy.
7. Внедрить праздничный модуль: модель -> источник -> кеш -> применение -> UI.
8. Подготовить release checklist.
9. После стабилизации ядра вернуться к UI/UX.
```

## Главное правило дальнейшей разработки

Работать маленькими контролируемыми шагами.

Для каждого шага желательно иметь:

```text
- понятную проблему;
- минимальное изменение кода;
- unit/instrumentation-тест там, где это возможно;
- короткую документацию;
- ручной сценарий проверки, если затронуто Android-поведение;
- отдельный коммит с русским сообщением.
```

Так проект можно довести до production-уровня без хаотичных больших переписываний.
