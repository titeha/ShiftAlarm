# Первичный технический аудит ShiftAlarm

Дата анализа: 2026-07-08

Репозиторий: `titeha/ShiftAlarm`, ветка `master` на момент просмотра.

Статус проекта на момент анализа: незавершённый Android-проект, предрелизное состояние.

Фокус анализа: архитектура, движок расписаний, надёжность срабатывания будильника, сохранность данных, безопасность входов, память, производительность, тесты и документация. UI в этом документе сознательно не рассматривается как основной приоритет: интерфейс ещё будет меняться, а ядро и системная часть должны быть приведены в устойчивое состояние раньше.

Контекст от автора: проект создавался с помощью Claude Code, без предварительного полноценного архитектурного и риск-анализа. Работа велась итеративно: начали делать функциональность сразу, затем наращивали сценарии.

Анализ подготовил: ChatGPT (GPT-5.5 Pro).

## Краткий вывод

Проект выглядит не как случайный черновик, а как уже достаточно собранный предрелиз. Видна осмысленная предметная модель: сменные графики, циклы, отпуск, больничный, правки календаря, производственный календарь и реальные ограничения Android вокруг точных будильников.

Главная ценность проекта — не UI, а доменный движок смен и надёжная интеграция с Android `AlarmManager`. Это правильное направление: для пользователя такого приложения критично не то, насколько красив редактор, а то, зазвонит ли будильник в нужный день и время после Doze, перезагрузки, обновления приложения, изменения часового пояса, отпуска, больничного или ручной правки смены.

Архитектурно проект в целом заложен правильно: чистая логика расписаний отделена от Android/I/O, данные вынесены в Room, планирование через `AlarmManager` вынесено в отдельный слой, UI построен на Compose. Но перед дальнейшим развитием функций и UI нужно закрыть несколько базовых рисков в ядре и системной части.

Критичные направления:

1. Исключить ложные срабатывания будильника из устаревших или некорректных `PendingIntent`.
2. Исправить риск потери связанных данных из-за `@Insert(onConflict = REPLACE)` при наличии `ON DELETE CASCADE`.
3. Валидировать внешние intent-входы, особенно `ACTION_SET_ALARM`.
4. Сделать чтение `cycleSpec`, overrides и других строковых enum-полей устойчивым к повреждённым данным.
5. Достроить обработку системных событий Android: reboot, package replaced, time changed, timezone changed, разрешения точных будильников и уведомлений.
6. Добавить CI и явную матрицу ручного тестирования на устройствах.

## Откуда растёт проект

Проект держится на двух основаниях.

Первое — предметная область сменной работы. В README уже описаны сценарии, которые редко нормально покрываются обычными будильниками: графики 2/2, сутки/трое, утро/день/ночь, индивидуальный цикл, отпуск, больничный, отгул, правки календаря, переносы, производственный календарь. Это сильная ниша: приложение решает конкретную боль, а не пытается быть универсальным календарём.

Второе — системные ограничения Android. Надёжный будильник на Android — это не просто таймер. Нужно учитывать точные будильники, Doze, foreground service, notification/full-screen intent, перезагрузку, разрешения, OEM-ограничения, изменение времени и часового пояса. Проект уже идёт в правильную сторону, потому что использует `AlarmManager.setAlarmClock` для пользовательски видимого будильника и foreground service для звонка.

## Что уже хорошо

### Архитектурное разделение

В проекте есть понятное разделение слоёв:

- `schedule/` — чистая Kotlin-логика расписаний, смен, периодов, праздников и правок календаря;
- `data/` — Room-сущности, DAO, репозиторий и миграции;
- `alarm/` — интеграция с Android `AlarmManager`, `BroadcastReceiver`, foreground service и экраном звонка;
- `ui/` и `MainActivity` — Compose-интерфейс.

Это правильная структура. Движок расписаний можно тестировать без Android-устройства и без эмулятора. Android-слой остаётся тонким мостом к системе.

### Чистый движок расписаний

Фокус тестов уже сделан на `schedule/`: смены, отпуска, больничные, производственный календарь, custom cycle, overrides, расчёт следующего срабатывания. Это верное направление. Для такого приложения ошибки движка опаснее, чем шероховатости UI.

### Миграции БД

Room настроен с `exportSchema = true`, миграции описаны явно, `fallbackToDestructiveMigration` не используется. Это хорошо: пользовательские будильники, периоды и правки календаря нельзя терять при обновлении приложения.

### Минимальная поверхность атаки

В проекте нет сетевой части, серверного API, выполнения команд, десериализации недоверенного бинарного формата, WebView или сложной IPC-модели. Основные входы — Android intents, Room-БД и пользовательские данные из UI. Поэтому риски RCE/инъекций невысокие. Основные реальные риски — crash/DoS, ложный звонок, несрабатывание звонка или потеря расписания.

### Выбор `setAlarmClock`

Для будильника это правильный системный механизм. Он предназначен для пользовательски видимых, точных и критичных по времени сигналов. Но вокруг него ещё нужно достроить обработку разрешений, reboot/package/time events и деградации на разных OEM-прошивках.

## Найденные проблемы и риски

Приоритеты:

- P0 — исправить до любых публичных сборок, где пользователь может реально полагаться на будильник;
- P1 — исправить до beta/широкого тестирования;
- P2 — желательно исправить, но не блокирует раннюю внутреннюю разработку;
- P3 — косметика, поддерживаемость, будущие улучшения.

### P0. `AlarmReceiver` запускает звонок до проверки будильника

Файл: `app/src/main/java/ru/titeha/shiftalarm/alarm/AlarmReceiver.kt`.

Сейчас `AlarmReceiver.onReceive()` сначала вызывает `AlarmService.start(context)`, а только потом читает `alarm_id`, проверяет `NO_ID`, открывает БД и ищет будильник.

Риск: если прилетит устаревший `PendingIntent`, некорректный intent без id, intent от уже удалённого будильника или будильник уже выключен/изменён, сервис всё равно успеет начать звонить. Затем код может выйти из `runBlocking`, не остановив уже запущенный сервис.

Практический эффект: ложный звонок. Для будильника это критичный дефект.

Что нужно сделать позже:

- читать и валидировать `alarm_id` до запуска сервиса;
- подгружать будильник из БД до старта звука;
- проверять, что будильник существует и включён;
- желательно проверять, что срабатывание соответствует актуальной версии/поколению будильника, а не старому `PendingIntent`;
- запускать `AlarmService` только после успешной проверки;
- передавать `alarm_id` в сервис, чтобы сервис не был безымянным;
- добавить тесты/сценарии: `NO_ID`, удалённый будильник, выключенный будильник, изменённый будильник, повторное срабатывание старого intent.

### P0. `@Insert(onConflict = REPLACE)` может удалять связанные периоды и правки

Файл: `app/src/main/java/ru/titeha/shiftalarm/data/AlarmDao.kt`.

Сейчас DAO использует:

```kotlin
@Insert(onConflict = OnConflictStrategy.REPLACE)
suspend fun upsert(alarm: AlarmEntity): Long
```

В SQLite стратегия `REPLACE` при конфликте по `PRIMARY KEY`/`UNIQUE` удаляет старую строку перед вставкой новой. При этом в базе есть связанные таблицы `alarm_periods` и `alarm_overrides` с внешним ключом на `alarms.id` и `ON DELETE CASCADE`.

Риск: обновление существующего будильника через `REPLACE` может удалить его связанные периоды и правки календаря. Особенно опасный сценарий — обычное включение/выключение будильника из списка. В коде рядом есть комментарий, что периоды при этом не трогаются, но фактическая реализация через `repo.upsert(alarm)` может привести к каскадному удалению.

Практический эффект: пользователь меняет флаг `enabled`, а приложение может потерять отпуск, больничный, отгулы и overrides для этого будильника.

Что нужно сделать позже:

- не использовать `REPLACE` для обновления существующего `AlarmEntity`;
- разделить insert и update;
- либо использовать Room `@Upsert`, если версия Room и желаемая семантика подходят;
- либо реализовать `insertOrIgnore + update` внутри `@Transaction`;
- отдельно проверить, что id существующего будильника сохраняется;
- добавить тест: создать будильник, добавить periods/overrides, переключить `enabled`, убедиться, что связанные записи не исчезли.

### P1. Внешний `ACTION_SET_ALARM` требует жёсткой валидации

Файлы:

- `app/src/main/AndroidManifest.xml`;
- `SetAlarmActivity` / `SetAlarmSpec`.

`SetAlarmActivity` экспортирована и обрабатывает `android.intent.action.SET_ALARM`. Это ожидаемо для интеграции с Ассистентом, часами и сторонними приложениями. Но экспортированная точка входа должна считать intent недоверенным.

Риск: если извлечённые из intent час/минута/дни/название не валидируются строго, можно записать некорректный будильник в БД или позже получить crash в расчётах `LocalTime.of(...)`, `AlarmTimes.next(...)` и scheduler.

Что нужно сделать позже:

- валидировать `hour in 0..23`;
- валидировать `minute in 0..59`;
- нормализовать/ограничить длину label/message;
- валидировать days mask;
- для некорректного intent завершать activity без записи в БД;
- не допускать crash по внешнему intent;
- добавить unit/Robolectric-тесты на битые intent extras.

### P1. Чтение `cycleSpec` и строковых enum-полей не должно ронять приложение

Файлы:

- `app/src/main/java/ru/titeha/shiftalarm/schedule/ShiftCycleCodec.kt`;
- код преобразования `AlarmOverride.toDayOverride()`;
- места, где вызывается `ShiftCycleCodec.decode(...)`.

`ShiftCycleCodec.decode()` сейчас явно документирован как функция, которая бросает исключение на несовместимом вводе. Это допустимо для внутреннего чистого парсера, но опасно на границе БД/миграций/backup-restore. Аналогично строковые enum-поля вроде `category` и `polarity` нельзя читать через небезопасный `valueOf` без защитного слоя.

Риск: повреждённая БД, старая версия схемы, ручной restore, баг миграции или некорректная строка могут уронить расчёт расписания, список будильников или массовое перепланирование.

Практический эффект: один повреждённый будильник может сломать работу всего приложения или всех будущих срабатываний.

Что нужно сделать позже:

- оставить строгий `decode()` для тестов/внутренних гарантий, но добавить безопасный `decodeOrNull()` / `decodeResult()`;
- на границе БД использовать безопасный парсинг;
- повреждённый будильник не должен ронять весь список/рескейджулинг;
- в UI показывать такой будильник как требующий исправления;
- добавить тесты на битый `cycleSpec`, неизвестную категорию override, некорректное время в цикле.

### P1. Недостаточно системных событий для перепланирования

Файл: `app/src/main/java/ru/titeha/shiftalarm/alarm/BootReceiver.kt`.

Сейчас обрабатывается `BOOT_COMPLETED`. Это необходимый минимум, потому что Android сбрасывает зарегистрированные alarm-задания после выключения/перезагрузки устройства. Но для надёжного будильника этого мало.

Что нужно учесть позже:

- `MY_PACKAGE_REPLACED` — после обновления приложения;
- `TIME_CHANGED` — пользователь вручную изменил время;
- `TIMEZONE_CHANGED` — изменился часовой пояс;
- возможно `LOCKED_BOOT_COMPLETED` и direct boot, если появится задача звонить до первого unlock после перезагрузки;
- изменение/отзыв разрешения точных будильников;
- изменение разрешения уведомлений и full-screen intent.

Практический эффект текущего состояния: после части системных событий ближайший будильник может остаться неактуально запланированным или не быть запланирован вовсе.

### P1. `AlarmService` не имеет полноценного fallback, если звук недоступен

Файл: `app/src/main/java/ru/titeha/shiftalarm/alarm/AlarmService.kt`.

Сейчас `startRinging()` пытается получить системный alarm/ringtone URI и запустить `MediaPlayer`. Если это не получилось, исключение проглатывается, а сервис оставляет уведомление/экран.

Риск: будильник “сработал”, но без звука. Для обычного приложения это приемлемая деградация, для будильника — недостаточно.

Что нужно сделать позже:

- добавить fallback на вибрацию;
- добавить fallback на другой системный звук;
- возможно добавить простой встроенный generated tone или bundled raw resource;
- логировать/фиксировать факт деградации;
- проверить поведение без выбранного системного alarm sound.

### P1. Разрешения Android нужно сделать частью продукта, а не только manifest

Файл: `app/src/main/AndroidManifest.xml`.

Manifest уже содержит разрешения для точных будильников, уведомлений, full-screen intent, foreground service, reboot и вибрации. Но наличие разрешений в manifest не гарантирует, что пользователь или система реально разрешили все нужные возможности.

Что нужно сделать позже:

- явно проверять возможность ставить exact alarms;
- показывать пользователю понятное состояние: “точные будильники разрешены/запрещены”;
- проверять уведомления;
- проверять full-screen intent / heads-up сценарии на актуальных Android;
- давать переход в настройки, если критичное разрешение выключено;
- добавить экран/диагностику “готовность будильника”.

### P1. `runBlocking` в `BroadcastReceiver` может блокировать поток при работе с БД

Файлы:

- `AlarmReceiver.kt`;
- `BootReceiver.kt`.

Использование `goAsync()` само по себе правильно, но внутри всё равно выполняется `runBlocking`. Сейчас объёмы маленькие, поэтому практический риск невелик. Но с ростом логики перепланирования, БД и проверок лучше не держать broadcast thread заблокированным.

Что нужно сделать позже:

- запускать работу через coroutine scope на `Dispatchers.IO`;
- завершать `pending.finish()` в `finally` после асинхронной работы;
- ограничить время операции;
- сделать обработку исключений явной, чтобы один сбой БД не оставил неопределённое состояние.

### P2. `AlarmScheduler.rescheduleAll` делает N+1 обращения к БД

Файл: `app/src/main/java/ru/titeha/shiftalarm/alarm/AlarmScheduler.kt`.

`rescheduleAll()` проходит по списку включённых будильников и для каждого отдельно подгружает периоды и overrides. Для обычного пользователя с несколькими будильниками это нормально. Но архитектурно это N+1-паттерн.

Что можно сделать позже:

- одним запросом получить все periods для списка alarmId;
- одним запросом получить все overrides для списка alarmId;
- сгруппировать их в памяти;
- затем перепланировать будильники без дополнительных запросов на каждый элемент.

Это не срочный performance-баг, но хорошая точка для оптимизации перед большим количеством расписаний или импортом.

### P2. Расчёт `next` в Compose-строке списка выполняется во время композиции

Файл: `app/src/main/java/ru/titeha/shiftalarm/MainActivity.kt`.

В `AlarmRow` во время композиции вызывается `AlarmTimes.next(...)`. Для маленького списка это приемлемо. Но при частых recomposition и большом количестве будильников расчёт лучше вынести в state/ViewModel-слой или кэшировать.

Так как UI сейчас не приоритет, это можно оставить на потом. Важно только помнить, что расчёты расписания лучше не размазывать по UI.

### P2. `allowBackup="true"` нужно осознанно подтвердить или изменить

Файл: `app/src/main/AndroidManifest.xml`.

Сейчас backup разрешён. Это может быть нормально, потому что пользователь не хочет терять будильники при переносе устройства. Но данные о сменах, отпусках, больничных и рабочем графике могут считаться личными. Кроме того, после restore на новом устройстве нужны повторные проверки разрешений и перепланирование.

Что нужно решить позже:

- оставить backup включённым и корректно обрабатывать restore;
- либо ограничить backup rules;
- либо выключить backup до появления нормальной restore-логики;
- задокументировать решение.

### P2. JitPack-зависимость — supply-chain-точка

Проект использует отдельный компонент `AnalogTimePicker`, подключённый через JitPack. Это не проблема само по себе, особенно если компонент свой. Но для reproducible build и будущей публикации нужно понимать, что сборка зависит от внешнего JitPack/GitHub артефакта.

Что нужно сделать позже:

- фиксировать версию зависимости;
- не использовать плавающие версии;
- добавить dependency verification/lock, если проект пойдёт в публичную публикацию;
- иметь fallback-план, если JitPack недоступен.

### P2. Нет видимого CI workflow

На момент анализа в репозитории не было явно настроенного GitHub Actions workflow для регулярного `testDebugUnitTest` на push/pull request.

Что нужно сделать позже:

- добавить CI на `push` и `pull_request`;
- запускать минимум `./gradlew testDebugUnitTest`;
- по возможности добавить `lint`;
- миграционные instrumented-тесты оставить отдельным ручным или nightly workflow, потому что нужен Android device/emulator;
- хранить отчёты тестов как artifacts при падении.

## Производительность и память

Сейчас явных критичных проблем по производительности не видно. Типичный пользователь будет иметь единицы или десятки будильников, а не тысячи. Большинство циклов в движке ограничены небольшим горизонтом поиска: ближайшие дни/месяц/год. Для обычной нагрузки это нормально.

Основные потенциальные узкие места:

- N+1-запросы в `rescheduleAll()`;
- пересчёт `AlarmTimes.next()` внутри Compose-композиции;
- проверка отпусков/периодов при freeze-cycle через проход по периодам;
- парсинг custom cycle из строки в местах, где это можно кэшировать;
- отсутствие пакетной загрузки periods/overrides при массовом перепланировании.

Классических утечек памяти на статическом просмотре не видно. Нет unmanaged resources вне `MediaPlayer`; `MediaPlayer` освобождается в `stopRingingAndSelf()` и `onDestroy()`. Но нужно аккуратно доработать сценарии ошибок в `startRinging()`, чтобы частично созданный `MediaPlayer` не оставался в неопределённом состоянии при исключении между `MediaPlayer()` и `start()`.

## Безопасность

Основная модель угроз для этого проекта — не удалённый взлом, а устойчивость к некорректным входам и системным условиям:

- внешний `ACTION_SET_ALARM`;
- повреждённая или восстановленная БД;
- устаревшие `PendingIntent`;
- отозванные разрешения;
- перезагрузка/обновление приложения/смена времени;
- OEM-ограничения foreground/full-screen intent;
- недоступный звук будильника.

Сильные стороны:

- большинство внутренних компонентов не экспортируется;
- нет сети;
- нет WebView;
- нет небезопасной десериализации;
- нет запуска внешних команд.

Основные меры, которые нужны дальше:

- defensive parsing на границе БД;
- строгая валидация exported intent-входов;
- безопасная семантика обновления Room-сущностей;
- проверка актуальности alarm-события перед запуском звука;
- явная диагностика разрешений.

## Тесты, которые стоит добавить первыми

### Данные и Room

### AlarmReceiver / Scheduler

1. Intent без `alarm_id` не запускает `AlarmService`.
2. Intent с id удалённого будильника не запускает `AlarmService`.
3. Intent с выключенным будильником не запускает `AlarmService`.

5. Производственный календарь проверяется на переносах, праздниках и разных polarity.

### Внешние intents

1. `ACTION_SET_ALARM` с `hour = -1` / `hour = 24` не создаёт будильник.
2. `ACTION_SET_ALARM` с `minute = -1` / `minute = 60` не создаёт будильник.
3. Слишком длинный label нормализуется или отклоняется.
4. Пустой/битый набор дней не приводит к crash.

## Архитектурные инварианты, которые стоит закрепить

Эти правила лучше явно держать в документации и тестах:

1. Звук будильника нельзя запускать до проверки актуальности события.
2. Обновление `AlarmEntity` не должно удалять periods/overrides, если пользователь явно не редактирует эти списки.
3. Один повреждённый будильник не должен ломать все остальные будильники.
4. Движок расписаний не должен зависеть от Android API, времени системы напрямую и UI.
5. Любая внешняя точка входа должна валидировать данные как недоверенные.
6. После системных событий, влияющих на время или регистрацию alarms, приложение должно перепланировать актуальные будильники.
7. Миграции БД должны сохранять пользовательские данные; destructive migration недопустима без отдельного решения.
8. UI может меняться, но контракт движка и данных должен быть стабильным и покрытым тестами.
