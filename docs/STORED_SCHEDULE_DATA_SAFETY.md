# Защитное чтение сохранённых данных расписания

## Назначение документа

Этот документ фиксирует правило чтения сохранённых данных расписания из БД.

ShiftAlarm хранит пользовательские настройки графика в нескольких полях:

```text
alarms.cycleSpec
alarm_overrides.category
alarm_overrides.wakeMinutes
alarm_overrides.fromEpochDay
alarm_overrides.toEpochDay
```

Эти данные обычно создаёт само приложение, но их всё равно нельзя считать абсолютно надёжными.

## Почему нужна защита

Данные могут стать некорректными после:

```text
ошибки старой версии приложения;
незавершённой миграции;
backup/restore между версиями;
ручного восстановления БД;
будущего изменения формата cycleSpec;
ошибки в UI или импорте данных.
```

Если чтение таких данных бросает исключение, может упасть:

```text
планирование будильника;
перепланирование после загрузки;
AlarmReceiver при срабатывании;
экран календаря смен;
список будильников.
```

Для будильника это недопустимо: одна повреждённая запись не должна ломать весь планировщик.

## cycleSpec

cycleSpec хранит пользовательский цикл смен в строковом формате.

Для строгого разбора используется:

```code
ShiftCycleCodec.decode
```

Этот метод может бросить исключение на несовместимом формате.

Для чтения из БД используется безопасный метод:

```code
ShiftCycleCodec.decodeOrNull
```

Если строка повреждена, метод возвращает null.

## Поведение при повреждённом cycleSpec

Если у будильника повреждён пользовательский цикл, расчёт пытается откатиться на встроенный пресет.

Правило:

```text
cycleSpec корректен и непустой -> использовать пользовательский цикл;
cycleSpec пустой или повреждён -> использовать presetId;
presetId неизвестен -> не планировать будильник.
```

Такой подход сохраняет работоспособность приложения и не создаёт неожиданный график из повреждённых данных.

## alarm_overrides

Правки календаря разворачиваются через:

```code
AlarmOverride.toDayOverrideOrNull
```

Метод возвращает null, если запись повреждена:

```text
категория не входит в ShiftCategory;
wakeMinutes меньше 0;
wakeMinutes больше 1439;
дата не преобразуется из epoch day;
конец диапазона раньше начала.
```

Повреждённые правки пропускаются.

## Почему повреждённые правки пропускаются

Правка календаря уточняет базовый график, но не является единственным источником расписания.

Если одна правка повреждена, безопаснее пропустить её, чем уронить:

```text
AlarmReceiver;
BootReceiver;
AlarmScheduler;
экран календаря.
```

Базовый график при этом продолжит работать.

## Что проверяется тестами

Unit-тесты проверяют:

```text
decodeOrNull возвращает null для битого cycleSpec;
decodeOrNull сохраняет поведение для пустой строки;
битый cycleSpec откатывается на пресет;
битый cycleSpec без пресета не планируется;
корректная AlarmOverride разворачивается в DayOverride;
правки с некорректной категорией, временем и диапазоном пропускаются.
```

## Правило для будущих изменений

Любые данные, прочитанные из БД и влияющие на планирование будильника, должны иметь безопасный путь разбора.

Если новый формат может быть повреждён или устареть, нужен метод вида:

```text
parseOrNull
decodeOrNull
toDomainOrNull
```

И unit-тесты на повреждённые данные.
