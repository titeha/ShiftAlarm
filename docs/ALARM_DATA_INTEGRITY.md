# Сохранность связанных данных будильника

## Назначение документа

Этот документ фиксирует правило сохранения будильников в ShiftAlarm.

Правило важно для сохранности пользовательских данных: периодов без будильника, правок календаря, выходных, отгулов, больничных и других уточнений расписания.

## Проблема

В ранней реализации обновление будильника выполнялось через `@Insert(onConflict = OnConflictStrategy.REPLACE)`.

Для таблицы `alarms` это небезопасно.

В SQLite поведение `REPLACE` фактически означает удаление старой строки при конфликте и вставку новой строки. Если к этой строке привязаны дочерние записи через `ON DELETE CASCADE`, они могут быть удалены вместе со старой строкой.

В ShiftAlarm к будильнику привязаны:

```text
alarm_periods
alarm_overrides
```

Эти таблицы используют внешний ключ на `alarms.id` с каскадным удалением. Поэтому обновление существующего будильника через `REPLACE` могло привести к потере связанных периодов и правок календаря.

## Пример риска

Пользователь создал будильник для сменного графика.

Затем добавил к нему:

```text
период без будильника;
ручную правку календаря;
выходной или отгул.
```

После этого пользователь изменил сам будильник, например:

```text
переименовал;
включил или выключил;
изменил время;
изменил параметры графика.
```

Если такое обновление выполнялось через `REPLACE`, связанные записи могли быть удалены каскадно.

Для пользователя это выглядело бы как внезапная потеря настроек календаря.

## Принятое правило

Для таблицы `alarms` нельзя использовать `REPLACE` при обновлении существующей записи.

Сохранение будильника должно выполняться так:

```text
если будильник новый — INSERT;
если будильник уже существует — UPDATE.
```

Обновление через `UPDATE` сохраняет ту же строку в таблице `alarms`, поэтому связанные периоды и правки календаря не удаляются.

## Текущее поведение

Метод сохранения будильника работает по правилу:

```text
id == 0L  -> insert
id != 0L  -> update
```

Название метода `upsert` может быть сохранено для совместимости с существующими вызовами, но внутри он не должен использовать SQLite `REPLACE`.

## Каскадное удаление

Каскадное удаление остаётся правильным поведением только при явном удалении будильника.

Если пользователь удаляет будильник, связанные с ним данные должны быть удалены:

```text
периоды без будильника;
правки календаря;
переопределения смен.
```

Это ожидаемое поведение, потому что связанные записи без родительского будильника не имеют смысла.

## Что проверяется тестами

Добавлен инструментальный Room-тест, который проверяет два сценария.

Первый сценарий:

```text
создать будильник;
добавить период без будильника;
добавить правку календаря;
обновить сам будильник через update;
проверить, что период и правка остались в базе.
```

Второй сценарий:

```text
создать будильник;
добавить период без будильника;
добавить правку календаря;
удалить будильник;
проверить, что связанные записи удалены каскадно.
```

Так тест фиксирует оба правила:

```text
обновление не удаляет связанные данные;
явное удаление удаляет связанные данные.
```

## Практическое значение

Это исправление защищает пользовательские настройки от тихой потери при обычном редактировании будильника.

Для приложения-будильника это критично, потому что пользователь может заранее настроить отпуска, больничные, отгулы, праздничные дни и ручные исключения. Эти данные не должны исчезать при переименовании будильника или изменении базовых параметров.

## Правило для будущих изменений

При работе с таблицей `alarms` нужно соблюдать правило:

```text
не использовать REPLACE для обновления существующего будильника.
```

Если в будущем появятся новые дочерние таблицы, связанные с `alarms.id`, это правило останется актуальным.

Для новых дочерних таблиц нужно отдельно проверить:

```text
что происходит при обновлении будильника;
что происходит при удалении будильника;
нужен ли ON DELETE CASCADE;
покрыто ли поведение тестом.
```
