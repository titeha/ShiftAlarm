# Безопасная обработка срабатывания будильника

## Назначение документа

Этот документ фиксирует правило обработки системного срабатывания будильника в `AlarmReceiver`.

Правило важно для надёжности приложения: будильник не должен начинать звонить, если пришёл некорректный, устаревший или уже неактуальный `PendingIntent`.

## Проблема

Ранний вариант `AlarmReceiver` запускал `AlarmService` сразу при входе в `onReceive`.

Это опасно, потому что до запуска звонка нужно проверить:

```text
есть ли в intent корректный alarm_id;
существует ли такой будильник в базе;
включён ли этот будильник;
соответствует ли пришедшее срабатывание текущему расписанию будильника.
```

Если запустить сервис до этих проверок, возможен ложный звонок.

## Пример ложного срабатывания

Пользователь изменил время будильника с 07:00 на 08:00.

Старый `PendingIntent` на 07:00 по какой-то причине всё ещё был доставлен системой.

Если ресивер сразу запускает сервис, будильник зазвонит в 07:00, хотя пользователь уже изменил его на 08:00.

## Принятое правило

`AlarmReceiver` не запускает `AlarmService` сразу.

Порядок обработки должен быть таким:

```text
прочитать alarm_id из intent;
если alarm_id отсутствует — выйти;
загрузить будильник из базы;
если будильник удалён — выйти;
загрузить периоды без будильника и правки календаря;
проверить актуальность срабатывания через AlarmFireValidator;
если срабатывание устарело — перепланировать будильник и выйти;
если срабатывание актуально — запустить AlarmService;
после запуска обработать судьбу будильника: удалить, выключить или перепланировать.
```

## Плановое время в PendingIntent

При планировании будильника в intent записываются два значения:

```text
alarm_id
trigger_at_millis
```

`alarm_id` нужен для поиска будильника в базе.

`trigger_at_millis` нужен для проверки, что пришедшее срабатывание соответствует текущему расписанию будильника.

## Проверка актуальности

Проверка вынесена в `AlarmFireValidator`.

Валидатор не зависит от Android API. Он принимает текущую запись будильника, периоды, правки календаря и плановое время срабатывания.

Если будильник выключен, звонок запрещён.

Если плановое время отсутствует, intent считается intent старого формата. Для совместимости он разрешается только для включённого будильника.

Если плановое время есть, валидатор заново считает ожидаемое ближайшее срабатывание по текущему состоянию будильника и сравнивает его с временем из intent.

## Что проверяется тестами

Unit-тесты проверяют:

```text
включённый будильник с актуальным временем можно запускать;
выключенный будильник нельзя запускать;
устаревшее время срабатывания нельзя запускать;
intent старого формата разрешён для включённого будильника;
intent старого формата запрещён для выключенного будильника.
```

## Ограничения

Проверка не заменяет перепланирование после системных событий.

Отдельно нужно обработать:

```text
перезагрузку устройства;
обновление приложения;
изменение времени;
изменение часового пояса;
изменение разрешений на точные будильники.
```

Эти задачи относятся к отдельному этапу стабилизации Android-интеграции.

## Правило для будущих изменений

В `AlarmReceiver` нельзя запускать `AlarmService` до проверки актуальности срабатывания.

Любая новая логика обработки intent должна сохранять порядок:

```text
сначала проверка;
потом запуск звонка;
затем удаление, выключение или перепланирование будильника.
```
