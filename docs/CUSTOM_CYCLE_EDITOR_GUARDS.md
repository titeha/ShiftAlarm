# Ограничения операций редактора пользовательского цикла

## Назначение документа

Этот документ фиксирует правила непосредственного редактирования пользовательского сменного цикла.

Итоговый `AlarmEditorValidator` остаётся последней защитой перед сохранением. Однако редактор не должен позволять пользователю выполнить действие, результат которого заранее нельзя сохранить.

## Общий лимит цикла

Пользовательский цикл после развёртки блоков может содержать не более:

```text
60 календарных дней
```

Один блок может содержать не более:

```text
30 одинаковых дней подряд
```

Ограничение проверяется при:

```text
добавлении нового блока;
увеличении существующего блока;
изменении размера нового блока;
дублировании существующего блока;
подтверждении изменений блока;
итоговом сохранении будильника.
```

## Редактирование существующего блока

При изменении блока его старый размер временно исключается из расчёта.

Пример:

```text
Общая длина цикла: 59 дней.
Редактируемый блок: 10 дней.
Остальные блоки: 49 дней.
```

Такой блок можно увеличить до 11 дней:

```text
49 + 11 = 60
```

Увеличение до 12 дней блокируется:

```text
49 + 12 = 61
```

## Дублирование блока

Дублирование учитывает локально изменённый размер блока.

Итоговая длина считается по формуле:

```text
остальные дни + изменённый блок + копия блока
```

Если итог превышает 60 дней, действие «Дублировать» недоступно.

## Новый блок

Нажатие «+ блок» больше не изменяет `cycleSpec` немедленно.

Сначала создаётся временный черновик нового блока и открывается bottom sheet.

Поведение:

```text
✓ — добавить подготовленный блок в цикл;
✕ — отменить добавление;
смахивание bottom sheet — отменить добавление.
```

Отмена не должна:

```text
увеличивать длину цикла;
добавлять автоматически созданный блок;
изменять cycleSpec;
помечать новый блок как сохранённый.
```

## Почему проверка выполняется дважды

Контролы редактора предотвращают ожидаемые ошибочные действия сразу.

`AlarmEditorValidator` повторно проверяет итоговый цикл перед сохранением. Это защищает от:

```text
ошибки UI;
старых или повреждённых данных;
будущих изменений редактора;
обхода одного из локальных ограничений.
```

## Проверка тестами

Обычные unit-тесты проверяют:

```text
добавление блока до общего предела;
запрет добавления после 60 дней;
максимальный размер существующего и нового блока;
увеличение ровно до общего лимита;
запрет увеличения сверх лимита;
дублирование ровно до 60 дней;
запрет дублирования сверх лимита;
учёт локально изменённого размера блока.
```

Отмена нового блока проверяется вручную на устройстве, потому что это взаимодействие состояния Compose и bottom sheet.

## Правило для будущих изменений

Новая операция над циклом должна:

```text
знать итоговое число календарных дней;
не изменять основной cycleSpec до подтверждения;
не полагаться только на итоговый валидатор;
иметь чистую проверяемую часть, если содержит арифметику или правила.
```
