Документ для Claude Code. Задача: **проверить** каждый пункт чек-листа по коду, заполнить
отчёт, затем **исправить** пробелы в порядке приоритетов. Надёжность срабатывания — главное
свойство продукта: пропущенный звонок для пользователя означает проспанную смену.

## 1. Что по README уже заявлено (аудит должен подтвердить кодом)

README утверждает, что реализовано:

- звонок через `AlarmManager.setAlarmClock` (Doze-стойкость, системная иконка);

- вибрация как запасной сигнал;
- проверка готовности (разрешения) и диагностический журнал событий.

Каждую заявку подтверди конкретным местом в коде (файл + строка) в отчёте.

## 2. Чек-лист P0 — «проспал = катастрофа»

Формат пункта: **Проверить** → **Готово, когда** → **Если нет**.

### P0-1. Право на точные будильники (API 31+)

- **Проверить:** в манифесте объявлены `SCHEDULE_EXACT_ALARM` (с `maxSdkVersion="32"`)
  и `USE_EXACT_ALARM` (API 33+, для будильников легально и не отзывается системой).
  В `AlarmScheduler` на API 31–32 перед планированием — `alarmManager.canScheduleExactAlarms()`.
  Экран готовности умеет отправлять в `Settings.ACTION_REQUEST_SCHEDULE_EXACT_ALARM`.
  Ресивер `ACTION_SCHEDULE_EXACT_ALARM_PERMISSION_STATE_CHANGED` перевыставляет будильники
  (README относит это к `SystemRescheduleActions` — подтвердить).
- **Готово, когда:** на 31–32 без разрешения приложение не молчит, а показывает явное
  предупреждение с кнопкой в настройки; на 33+ используется `USE_EXACT_ALARM`.
- **Если нет:** добавить объявления, runtime-проверку и пункт в экран готовности.

### P0-2. Полный набор системных событий перепланирования

- **Проверить:** ресиверы (или `SystemRescheduleActions`) покрывают все четыре события:
  `BOOT_COMPLETED`, `MY_PACKAGE_REPLACED`, `TIME_SET`, `TIMEZONE_CHANGED`. Атрибуты
  `exported`/фильтры корректны; все `PendingIntent` создаются с явным `FLAG_IMMUTABLE`
  (или обоснованным `FLAG_MUTABLE`).
- **Готово, когда:** каждое событие ведёт к полному пересчёту и `setAlarmClock` ближайшего
  звонка + записи в диагностический журнал.
- **Если нет:** добить недостающие интенты в существующий механизм.

### P0-3. Direct Boot — ребут ночью, телефон остался залоченным

Самый вероятный пробел. Обычный `BOOT_COMPLETED`-ресивер после автообновления системы
ночью **не поможет**: до первого ввода пин-кода хранилище приложения (credential-encrypted,
включая Room) недоступно — базу не прочитать, будильники не перевыставить, человек проспал.

- **Проверить:** есть ли у boot-ресивера `android:directBootAware="true"`, слушается ли
  `ACTION_LOCKED_BOOT_COMPLETED`, существует ли кэш ближайших звонков в
  `context.createDeviceProtectedStorageContext()`.
- **Готово, когда:**
  1. при каждом перепланировании в device-protected хранилище (DataStore/prefs) пишется
     срез «ближайшие звонки на 7 дней»: `{alarmId, triggerAtMillis, label}`;
  2. directBootAware-ресивер на `LOCKED_BOOT_COMPLETED` перевыставляет их **вслепую**,
     не обращаясь к Room;
  3. по `ACTION_USER_UNLOCKED` (или первому старту приложения) выполняется полный пересчёт
     из Room и обновление кэша;
  4. сериализация кэша покрыта юнит-тестом.
- **Каверза:** `PendingIntent` для locked-boot-перевыставления должен собираться только из
  данных кэша (класс ресивера + extras), без единого обращения к базе.

### P0-4. Экран звонка сквозь блокировку

- **Проверить:** уведомление звонка — `CATEGORY_ALARM`, канал `IMPORTANCE_HIGH`,
  `fullScreenIntent` задан; `USE_FULL_SCREEN_INTENT` в манифесте; `AlarmActivity`
  использует `setShowWhenLocked(true)` + `setTurnScreenOn(true)`; на API 34+ перед показом
  проверяется `notificationManager.canUseFullScreenIntent()`.
- **Готово, когда:** при **отозванном** разрешении full-screen intent звонок всё равно
  звучит из сервиса, а heads-up-уведомление позволяет открыть экран и остановить сигнал.
  Экран готовности предупреждает об отозванном разрешении и ведёт в
  `Settings.ACTION_MANAGE_APP_USE_FULL_SCREEN_INTENT`.
- **Если нет:** добавить проверку и путь деградации.

### P0-5. Foreground-сервис звонка на targetSdk 34+

- **Проверить:** у `AlarmService` в манифесте объявлен `android:foregroundServiceType`
  (для будильника подходящий вариант — `systemExempted`, он предусмотрен для приложений
  с правом точных будильников, запускаемых по alarm; альтернатива — `mediaPlayback`).
  Объявлены разрешения `FOREGROUND_SERVICE` + соответствующее типу
  (`FOREGROUND_SERVICE_SYSTEM_EXEMPTED` / `FOREGROUND_SERVICE_MEDIA_PLAYBACK`).
  `AlarmReceiver` сразу вызывает `startForegroundService`; между срабатыванием ресивера и
  `startForeground()` удерживается wake lock (или используется `goAsync()`); сервис держит
  `PARTIAL_WAKE_LOCK`, пока играет сигнал.
- **Готово, когда:** старт сервиса по будильнику не падает на 34+ и звук не обрывается
  при погашенном экране. Раз targetSdk = 36 — сверь behavior changes 35/36 по FGS и
  зафиксируй в отчёте всё, что их касается; при неоднозначности — пометь, не гадай.

### P0-6. Звук гарантированно слышен

- **Проверить:** проигрывание идёт с `AudioAttributes.USAGE_ALARM` (alarm-стрим: громкость
  будильника, работает в DND при разрешённых будильниках), зациклено; вибрация параллельно
  (по README есть); `POST_NOTIFICATIONS` на 33+ — обязательный шаг проверки готовности
  (без него нет ни full-screen-уведомления, ни кнопки «Стоп»).
- **Готово, когда:** при громкости будильника = 0 срабатывает вибрация, а экран готовности
  заранее предупреждает «громкость будильника на нуле» (громкость самовольно не поднимать).
- **Если нет:** поправить атрибуты аудио и добавить предупреждение в готовность.

## 3. Чек-лист P1 — пользователь может сам себе навредить

### P1-1. Вендорские киллеры фона

- **Проверить:** есть ли экран/раздел «Настроить телефон» с инструкциями по производителю.
- **Готово, когда:** по `Build.MANUFACTURER` (Xiaomi/HyperOS, Huawei, Oppo/Realme, Vivo,
  Meizu, Samsung) показывается инструкция: автозапуск + исключение из батарейной экономии;
  best-effort интенты вендорских настроек обёрнуты в try/catch с текстовым fallback;
  есть ссылка-справочник dontkillmyapp.com. Раздел доступен из проверки готовности.

### P1-2. Игнор батарейной оптимизации

- **Проверить:** `PowerManager.isIgnoringBatteryOptimizations()` и запрос
  `ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` как пункт готовности (страхует сервис
  звонка; сам `setAlarmClock` от Doze не зависит).
- Пометка: для Google Play это чувствительное разрешение (нужна обоснованность — у
  будильника она есть), для RuStore ограничений нет. Реализовать как опциональный пункт
  готовности, не как блокирующий.

### P1-3. Самотест

- **Готово, когда:** в настройках/готовности есть «Проверить будильник» — боевой прогон
  всего пути (Scheduler → Receiver → Service → экран) через ~60 секунд, с теми же
  компонентами, что и настоящий звонок.

### P1-4. План/факт срабатываний

- **Проверить:** пишет ли диагностический журнал пары «запланировано T → сработало T'».
- **Готово, когда:** при старте приложения обнаруживается пропущенный/сильно опоздавший
  звонок и показывается карточка «звонок не прозвенел» со ссылкой на проверку готовности.

### P1-5. Устойчивость AlarmReceiver

- **Проверить:** ресивер укладывается в лимит (~10 c), тяжёлое I/O — через `goAsync()`;
  следующий звонок перевыставляется сразу в момент срабатывания текущего (а не только
  после взаимодействия пользователя).

## 6. Ручная проверка на устройстве (выполняет человек, не агент)

1. `adb shell dumpsys alarm | grep ru.titeha` — ближайший звонок виден как AlarmClock.
2. Doze: завести звонок через 2–3 мин → `adb shell dumpsys deviceidle force-idle` → ждать.
3. Ребут залоченным: звонок через ~10 мин → перезагрузка → **не разблокировать** → ждать.
4. Обновление: звонок через ~10 мин → `adb install -r` свежей сборки → не открывая
   приложение, ждать.
5. Ручной перевод времени и часового пояса в настройках → проверить пересчёт.
6. Громкость будильника 0 → вибрация + предупреждение в готовности.
7. DND: с разрешёнными будильниками и без.
8. Android 14+: отозвать full-screen intent в «Особом доступе» → проверить деградацию.
9. Xiaomi/Huawei при наличии: убить приложение из «недавних» → ждать звонок.
