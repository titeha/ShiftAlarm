# Direct Boot: будильники после ночного ребута залоченным

## Проблема (P0-3 аудита надёжности)

AlarmManager не хранит будильники между перезагрузками — их перевыставляет `BootReceiver`
по `BOOT_COMPLETED`. Но `BOOT_COMPLETED` приходит **только после первой разблокировки**.
Если устройство ночью автообновилось/перезагрузилось и осталось залоченным, до ввода
пин-кода credential-encrypted хранилище (включая Room) недоступно — будильники не
перевыставить, и звонок на смену пропадает.

## Что сделано (слой A — перевыставление)

1. **Device-protected кэш ближайших звонков.** При каждом планировании
   (`AlarmScheduler.refreshDirectBootCache`) в device-protected хранилище
   (`createDeviceProtectedStorageContext`, доступно до разблокировки) пишется срез включённых
   будильников на 7 дней: `{alarmId, triggerAtMillis, label}` (`DirectBootAlarmStore` +
   версионированный кодек `DirectBootAlarmCacheCodec`, покрытый unit-тестом).
2. **`directBootAware`-ресивер.** `BootReceiver` объявлен `android:directBootAware="true"` и
   слушает `LOCKED_BOOT_COMPLETED`. На это событие — **слепое** перевыставление
   (`AlarmScheduler.reArmFromCache`): для каждой записи кэша `setAlarmClock`, без единого
   обращения к Room. PendingIntent собирается только из данных кэша (id + время).
3. **Полный пересчёт при разблокировке.** По `USER_UNLOCKED` (и обычным событиям
   `BOOT_COMPLETED`/`TIME_SET`/…) — полный пересчёт из Room + обновление кэша.
4. Сериализация кэша покрыта `DirectBootAlarmCacheCodecTest`.

Итог слоя A: после ночного ребута залоченным будильники **перевыставляются рано** и не
теряются как таймеры.

## Слой B — звонок точно в срок при залоченном устройстве (СДЕЛАНО, проверено)

Весь путь звонка помечен `directBootAware` и не трогает Room до разблокировки:

- `AlarmReceiver`/`AlarmService`/`AlarmActivity` — `android:directBootAware="true"`;
- `AlarmReceiver`: при `!UserManager.isUserUnlocked()` звонит по метке из RingCache
  (`handleAlarmLocked`), не трогая Room/журнал; перепланирование отложено до `USER_UNLOCKED`.
  Просроченный за время ребута сверх грейса (30 мин) не звонит;
- `reArmFromCache` тоже с грейсом (просроченный в пределах грейса ставится на «сейчас»);
- запись в журнал в сервисе под try/catch (CE недоступен залоченным); текст экрана — из extras/кэша;
- **звук с fallback (этап 3):** `AlarmService.startSound` пробует (1) системную мелодию будильника →
  (2) любой рингтон → (3) встроенный в APK `res/raw/fallback_alarm.wav`. Последний важен для Direct
  Boot: системная мелодия может лежать в CE и до разблокировки быть нечитаемой, а raw-ресурс из APK
  читается всегда. Если не завёлся ни один источник — остаётся вибрация.

### ★ Ключевой баг (найден на устройстве): `Application` не должен трогать CE до разблокировки

При старте процесса до разблокировки (ради `directBootAware`-ресивера) система первым делом
зовёт `Application.onCreate()`. Если он обращается к credential-encrypted хранилищу (Room,
обычные SharedPreferences — у нас это был `HolidayCalendarRepository`), процесс **падает на
onCreate**, и `BootReceiver` не отрабатывает вообще. Поэтому `ShiftAlarmApp.onCreate` при
`!isUserUnlocked()` **пропускает всю CE-инициализацию** и выходит; перевыставление делает
`BootReceiver`, полная инициализация — при старте после разблокировки. Без этого фикса Direct
Boot не работал (процесс стартовал и сразу умирал).

**Проверено на Poco F6 Pro (HyperOS), автозапуск включён:** будильник через ~5 мин → ребут →
телефон НЕ разблокирован → **звонок прозвенел на локскрине в назначенное время** (лог
`Direct Boot: будильники перевыставлены из кэша`, `dumpsys alarm` показал перевыставленный
AlarmClock до разблокировки).

## Ручная проверка (на устройстве)

1. Автозапуск ВКЛ, будильник через ~8 мин → перезагрузка → **не разблокировать** → должен
   зазвонить на локскрине в срок.
2. `adb shell dumpsys alarm | grep ru.titeha` сразу после ребута (до разблокировки) — увидеть
   перевыставленный AlarmClock; лог `BootReceiver: Direct Boot ... перевыставлены из кэша`.
3. **Автозапуск ВЫКЛ** — на MIUI/HyperOS boot-broadcast к приложению не доставляется вовсе
   (см. `VENDOR_SETUP.md`); подстрахует перепланирование при открытии приложения.
