# Разрешения и готовность будильника

## Зачем

На современном Android будильник может молча не сработать, если пользователь не дал нужные
разрешения или система ограничивает приложение. Приложение должно это заметить и явно подсказать,
что настроить, а не полагаться на удачу.

## Что проверяется

| Проблема | Условие | Почему важно |
|----------|---------|--------------|
| Точные будильники | Android 12+ (`AlarmManager.canScheduleExactAlarms()`) | Без разрешения звонок может опоздать или не сработать вовремя. |
| Уведомления | Android 13+ (`POST_NOTIFICATIONS`) | Без них экран/уведомление звонка может не показаться. |
| Энергосбережение | `PowerManager.isIgnoringBatteryOptimizations()` | Doze/агрессивные OEM могут задержать или убить звонок. |

## Как это подано пользователю

Вверху списка будильников — баннер «Будильник может не сработать» с перечнем актуальных проблем
(по приоритету) и кнопкой «Настроить» у каждой, ведущей в нужный системный экран. Если проблем нет —
баннера нет. Статус перечитывается при каждом возврате в приложение (`ON_RESUME`), поэтому после
настройки баннер обновляется сам.

## Слои

| Компонент | Слой | Роль |
|-----------|------|------|
| `alarm/AlarmReadiness` | чистая логика | По булевым статусам → список проблем (`AlarmReadinessIssue`). Покрыт юнит-тестами. |
| `alarm/AlarmPermissions` | Android | Чтение реальных статусов + интенты в настройки. |
| `ui/AlarmReadinessBanner` | UI | Баннер; обновляется на `ON_RESUME` через жизненный цикл Activity. |

## Заметки

```text
- энергосбережение открывается СПИСКОМ приложений (ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS),
  без прямого запроса-исключения — так безопаснее для политики Google Play;
- на MIUI/Xiaomi настройки энергосбережения и «автозапуска» спрятаны и неочевидны — в будущем
  можно добавить OEM-специфичную подсказку;
- точные будильники для приложений-будильников на Android 13+ выдаются автоматически
  (USE_EXACT_ALARM), но пользователь может отозвать — поэтому статус всё равно проверяется.
```
