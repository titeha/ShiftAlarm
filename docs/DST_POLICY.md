# Часовой пояс и переходы летнего времени (DST)

## Модель времени

Будильник задаётся в **локальном календарном времени** («каждый день в 07:00», «ночь,
подъём 05:00»). Поэтому весь расчёт расписания идёт в `LocalDate`/`LocalTime`/
`LocalDateTime` без привязки к абсолютному моменту.

В абсолютный момент (`Instant` → epoch millis для `AlarmManager`) локальное время
переводится **только на границе планирования**, по часовому поясу устройства. Это
осознанный выбор: хранить будильник как «замороженный» `Instant` нельзя — после смены
пояса он прозвонил бы не в 07:00 по местным часам.

Смену пояса и перевод часов подхватывает перепланирование: система шлёт
`TIMEZONE_CHANGED`/`TIME_SET`, и будильники пересчитываются (см. `SYSTEM_RESCHEDULE.md`).
То есть приложение **следует локальному времени устройства**.

## Единая точка перевода

Перевод локального времени в момент вынесен в чистый объект `schedule/AlarmInstant.kt`:

```text
AlarmInstant.of(local, zone): Instant
AlarmInstant.epochMilli(local, zone = пояс устройства): Long
```

`AlarmScheduler` использует только его, поэтому политика ниже применяется в одном месте
и покрыта unit-тестами.

## Политика неоднозначного локального времени

Дважды в год локальное время становится неоднозначным. Правило — следовать локальному
времени устройства:

```text
Пробел (перевод вперёд, локального времени нет, напр. 02:30):
  момент сдвигается ВПЕРЁД на длину пробела по правилам зоны;
  будильник не пропадает, а звонит сразу после перевода.

Наложение (перевод назад, локальное время дважды, напр. 02:30):
  берётся ПЕРВОЕ вхождение (более ранний offset);
  один будильник не звонит дважды.
```

Обе политики уже реализует `LocalDateTime.atZone` (gap → сдвиг вперёд, overlap → ранний
offset). `AlarmInstant` фиксирует это как явное поведение, а `AlarmInstantTest` проверяет
на реальных датах перехода (США, 2026): пробел 8 марта, наложение 1 ноября, а также
зависимость результата от зоны.

## Границы

```text
доменная модель на Instant НЕ переводится — будильник остаётся локальным;
пояс не хранится в записи будильника — используется текущий пояс устройства;
поездки: следуем времени устройства (перепланирование по TIMEZONE_CHANGED);
для РФ переходов DST сейчас нет, но политика важна до международного запуска.
```
