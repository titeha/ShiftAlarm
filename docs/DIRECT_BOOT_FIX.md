# Direct Boot: будильник после перезагрузки без разблокировки

Документ для Claude Code. **Расширяет и заменяет пункт P0-3** из `docs/RELIABILITY_AUDIT.md`
(в `RELIABILITY_REPORT.md` пометь P0-3: «реализуется по docs/DIRECT_BOOT_FIX.md»).

## Контекст и воспроизведение

Реальное устройство (Poco F6 Pro, HyperOS), автозапуск включён:
будильник заведён → перезагрузка → телефон **не разблокирован** → звонок не пришёл.

**Контрольный тест подтверждён владельцем**: тот же сценарий, но с разблокировкой
после ребута (приложение не открывалось) — звонок прозвенел штатно. Вывод:
автозапуск и весь `BOOT_COMPLETED`-путь исправны — **их не менять**. Единственная
дыра — заблокированное состояние, работать только над ней.

Это штатное поведение Android (File-Based Encryption), не вендорская особенность:

- до первого ввода пин-кода `BOOT_COMPLETED` **не рассылается вовсе** — он приходит
  только после разблокировки;
- до разблокировки рассылается только `LOCKED_BOOT_COMPLETED`, и получить его могут
  лишь компоненты с `android:directBootAware="true"`;
- компоненты без `directBootAware` **не могут быть запущены** до разблокировки —
  включая ресивер, на который наведён PendingIntent сработавшего будильника;
- credential-encrypted хранилище (Room, обычные prefs/файлы) до разблокировки нечитаемо.

Следствие: чинить нужно не только перевыставление, а **весь путь звонка** в
заблокированном состоянии.

## Этап 1. Кэш звонков в device-protected storage

- Новый слой `RingCache` поверх `context.createDeviceProtectedStorageContext()`:
  SharedPreferences или DataStore с файлом в DPS-каталоге. **Room в DPS не заводить.**
- Содержимое: версия формата `v`; список ближайших звонков на 7 дней вперёд:
  `{alarmId, triggerAtMillis, label}`; сохранённый `BOOT_COUNT` (для сторожа из
  `XIAOMI_BOOT_FIX.md` — используйте один и тот же файл).
- Запись: при каждом перепланировании (все `reason` из инструментации).
- Юнит-тест сериализации/десериализации, включая незнакомую версию формата.

## Этап 2. directBootAware для всего пути звонка

В манифесте пометить `android:directBootAware="true"`:

1. **Boot-ресивер**: добавить `ACTION_LOCKED_BOOT_COMPLETED` в фильтр. В обработчике:
   если `UserManager.isUserUnlocked() == false` — работать **только** из `RingCache`
   (никаких обращений к Room/репозиторию), перевыставить звонки через существующий
   `AlarmScheduler`-путь `setAlarmClock`. **Просроченный за время ребута звонок**
   (`triggerAtMillis` уже в прошлом): если опоздание в пределах грейс-окна
   (константа, предложение — 30 минут) — звонить немедленно; если больше —
   записать в журнал как `missed`, не звонить.
2. **AlarmReceiver** (срабатывание): до анлока собирать всё необходимое для звонка
   из extras/`RingCache`, не трогая базу.
3. **AlarmService** и **AlarmActivity**: должны стартовать и работать до анлока.
   Канал уведомлений: создание идемпотентно — продублировать создание канала в
   direct-boot ветке через DPS-контекст.
4. Общее правило: в каждом компоненте пути звонка ветвление по `isUserUnlocked()`;
   всё, что требует CE-хранилища, — только в разблокированной ветке.

## Этап 3. Звук и экран до разблокировки

- Пользовательская мелодия (`content://media/...`) до анлока недоступна. Порядок
  fallback: (1) выбранный URI, если читается; (2) системный будильник по умолчанию
  `RingtoneManager.getDefaultUri(TYPE_ALARM)`; (3) встроенный в APK raw-ресурс
  (добавить короткий сигнал в `res/raw`, если его нет) + вибрация.
- Плеер — с `AudioAttributes.USAGE_ALARM` (сверься с P0-6 аудита).
- Экран: `setShowWhenLocked(true)` + `setTurnScreenOn(true)` у `AlarmActivity`
  (сверься с P0-4); текст на экране до анлока — из extras/кэша, не из Room.

## Этап 4. Синхронизация после разблокировки

- Ресивер на `ACTION_USER_UNLOCKED` (плюс подстраховка: проверка при старте
  приложения): полный пересчёт из Room, перевыставление, обновление `RingCache`.
- Журнал: до анлока писать события в буфер в DPS; после анлока — переливать буфер
  в основной диагностический журнал с пометкой `direct_boot`.

## Этап 5. Тест-матрица (выполняет человек на Poco F6 Pro)

| # | Сценарий | Ожидание после фикса |
|---|----------|----------------------|
| 1 | Автозапуск ВКЛ → будильник через 10 мин → ребут → **не разблокировать** | Звонок звенит на локскрине; журнал: `locked_boot` (из буфера) |
| 2 | То же, но **разблокировать** через 2–3 мин, приложение не открывать | Звонок звенит; журнал: `locked_boot` + `user_unlocked`-сверка. Проходил и до фикса (по `BOOT_COMPLETED`-пути) — регрессионный контроль |
| 2а | Ребут за ~2 мин **до** звонка, не разблокировать | Звонок звенит сразу после загрузки (грейс-окно), журнал: `locked_boot` |
| 3 | Автозапуск ВЫКЛ → сценарий 1 | Если звонит — HyperOS пропускает `LOCKED_BOOT` без автозапуска (зафиксировать в отчёте); если нет — после открытия приложения срабатывает сторож `watchdog` |
| 4 | Пользовательская мелодия из памяти телефона → сценарий 1 | Звонит запасным сигналом (default/raw), не молчит |

## Ограничения

- Трогать только `alarm/`, манифест, ресурсы (raw-сигнал, строки). Движок `schedule/`,
  Room-схему, `cycleSpec` — не менять. Новых зависимостей не добавлять.
- Каждый этап — отдельный коммит; `testDebugUnitTest` и `assembleDebug` зелёные.
- Сборка (Windows): `JAVA_HOME="C:\Program Files\Android\Android Studio\jbr"
  GRADLE_USER_HOME="H:\gradlecache" ./gradlew.bat testDebugUnitTest assembleDebug --console=plain`.
- Если инструментация из `docs/XIAOMI_BOOT_FIX.md` (журнал + reason) ещё не сделана —
  сделай её первым коммитом, она нужна для тест-матрицы.
