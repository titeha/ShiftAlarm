# Учебный модуль: школа и вуз (чёт/нечёт) поверх сменного режима

Документ для Claude Code. Принцип: учёба — это **шаблон поверх существующего
MODE_SHIFT**, а не новый режим движка. Обычная учебная неделя = цикл на 7 дней,
чёт/нечёт вуза = цикл на 14 дней; оба привязаны к понедельнику. Каникулы и сессия =
существующие периоды. Движок `schedule/` не меняется ни на строку.

Рекомендуемый порядок: после `SNOOZE_AND_REPEAT.md` и `WORKWEEK_AND_HOLIDAYS.md`.
Зависимости мягкие: от WORKWEEK берутся типизированные праздники и порядок дней
недели по настройке «Начало недели»; механически модуль работает и без них.

---

## 0. Модель

### Школа / «одинаковая неделя»
- Цикл из 7 слотов, слот = подъём в HH:MM или «выходной».
- Тип учебных слотов — существующий «День» (новых типов слотов **не вводить**:
  это формат `cycleSpec` и цвета календаря). Выходные — тип «Выходной».
- Якорь: ближайший прошедший понедельник от даты создания. Вопросов пользователю
  не нужно — 7-дневный цикл сам выравнивается по неделям.
- Шестидневка (Пн–Сб) — просто заполненная суббота. Праздники глушат любой день
  цикла (HOLIDAY в сменном режиме уже работает через HolidayAlarms); переносы
  рабочих суббот к циклам не применяются by design — для школ это корректно.

### Вуз / «две недели» (чёт/нечёт)
- Цикл из 14 слотов = неделя А (слоты 0–6) + неделя Б (слоты 7–13).
- Якорь по вопросу пользователю: **«Какая неделя сейчас?»** [Нечётная] [Чётная].
  Вычислять чётность из номера календарной недели **запрещено** — вузы считают
  по-разному. Ответ «нечётная» → якорь = понедельник текущей недели;
  «чётная» → понедельник предыдущей.
- Границы недель цикла — всегда понедельник, независимо от настройки отображения
  «Начало недели» (она крутит только порядок колонок в UI).
- Нейминг пары недель — глобальная настройка приложения (не свойство будильника):
  «Чётная/Нечётная» (дефолт), «Верхняя/Нижняя», «Числитель/Знаменатель».
  Влияет только на подписи в мастере, календаре и превью.

### Каникулы и сессия
- Два новых значения `PeriodKind`: «Каникулы» и «Сессия» (+ цвета в палитру
  календаря, + строки). README фиксирует, что kind хранится в существующей колонке
  `reason` без миграции — **проверить и подтвердить в отчёте**, что новые значения
  действительно не требуют миграции и что неизвестный kind в старой версии
  приложения не роняет чтение (деградация в «период без типа»).
- Оба типа глушат звонок как обычный период; сессия — потому что расписание
  экзаменов хаотично: на экзамены пользователь ставит обычные разовые будильники.

### Совместная работа со сменным будильником (заочник)

Учебный будильник — отдельная сущность в общем списке; со сменным он не
конфликтует by design: у каждого свой цикл, периоды, подмены и календарь, движок
их не связывает. Сценарий заочника (работает 2/2, раз в полгода уезжает на месяц
сессии) собирается из готовых деталей **без нового кода**: рабочему будильнику —
период типа «Сессия» на месяц (типы периодов глобальны и доступны любому
будильнику), фаза цикла после возвращения — существующим переключателем
«Заморозить цикл на время отпуска»; учебный будильник на этот месяц включается
тумблером. Глобальный переключатель профиля «смены/учёба» **не вводить** — он
ломает ровно этот сценарий.

## 1. Что модуль НЕ делает (жёсткие рамки)

- Не моделирует пары, предметы, аудитории, звонки между уроками — мы будильник
  подъёма, а не расписание занятий.
- Не вводит новый режим в `AlarmEntity` и новые типы слотов в `cycleSpec`.
- Не перестраивает существующий редактор: точка входа — одна кнопка/чип «Учёба»
  в месте выбора пресетов сменного режима (агент находит место, вмешательство
  минимально). Повторное редактирование учебного будильника открывает **обычный**
  редактор цикла — полноценный учебный реэдит появится вместе с большим
  рефакторингом редактора (отдельное ТЗ).
- Смена расписания по четвертям/семестрам — правкой цикла или вторым будильником;
  автоматика — вне рамок, зафиксировать комментарием.

## 2. Мастер создания (новый экран, аддитивный)

Шаг 1. «Как устроена учебная неделя?» → [Одинаковая каждую неделю] [Две недели
чередуются]. Подписи второй опции — из настройки нейминга.

Шаг 2. Сетка времени подъёма:
- строки — дни недели, порядок по настройке «Начало недели»;
- для 14-дневного варианта — две колонки с подписями недель;
- тап по ячейке → выбор времени существующим `AnalogTimePicker` / «Выходной»;
- предзаполнение: Пн–Пт 07:00, Сб и Вс — выходной; у двухнедельного обе колонки
  предзаполнены одинаково;
- быстрое действие «Скопировать неделю А → Б» (для случая «отличается одна пара»).

Шаг 3 (только 14 дней). «Какая неделя сейчас?» — два крупных чипа с выбранными
названиями.

Сохранение: `StudyPlanBuilder` (см. §3) собирает слоты и якорь → создаётся обычный
сменный `AlarmEntity` (имя по умолчанию «Учёба»), дальше он живёт как любой сменный
будильник: календарь, подмены, периоды, снуз — всё наследуется бесплатно.

## 3. Чистая логика: `StudyPlanBuilder`

В `schedule/` или рядом, без Android:

```
buildStudyPlan(
    grid: List<LocalTime?>,        // 7 или 14 значений, null = выходной
    parityToday: Parity?,          // null для 7-дневного
    today: LocalDate
): StudyPlan(slots, anchorDate)
```

Юнит-тесты (обязательные):
1. 7 дней, создание в среду → якорь = понедельник этой недели; превью совпадает
   с сеткой.
2. 14 дней, «сейчас нечётная», создание в среду → сегодняшний слот = среда недели А.
3. 14 дней, «сейчас чётная» → якорь = понедельник прошлой недели; сегодняшний
   слот — из недели Б.
4. Создание в воскресенье (обоих вариантов) — без сдвига на единицу.
5. Переход через границу года / високосность — якорь корректен.
6. Пустая сетка (все выходные) — валидация: не давать сохранить, вернуть ошибку.

## 4. Этапы (коммиты)

- **Этап 0 — разведка**: где живут пресеты сменного режима и как встроить точку
  входа; подтвердить расширяемость `PeriodKind` без миграции; формат `cycleSpec`
  на 7/14 слотов. Короткий отчёт, без правок.
- **Этап 1 — `StudyPlanBuilder` + тесты** (§3). Чистый коммит без UI.
- **Этап 2 — мастер** (§2) + точка входа «Учёба».
- **Этап 3 — периоды «Каникулы» и «Сессия»** (§0): значения, цвета, строки,
  проверка обратной совместимости чтения.
- **Этап 4 (опционально) — бейдж текущей недели**: для 14-дневных циклов с якорем
  в понедельник показывать в календаре «сейчас: Нечётная» (эвристика
  `len == 14 && anchor.dayOfWeek == MONDAY`; зафиксировать комментарием, что это
  эвристика, а не флаг в данных).

## 5. Ручная проверка

1. Школьник, шестидневка Пн–Сб 07:30: суббота звонит; государственный праздник
   на субботе — тишина.
2. Студент, чёт/нечёт, создано в среду «нечётной»: превью «след:» и календарь
   показывают правильное чередование; через неделю — сменилось.
3. Каникулы диапазоном на две недели — тишина, календарь окрашен новым цветом.
4. Сессия + разовый будильник на экзамен внутри неё — разовый звонит, регулярка
   молчит.
5. «Начало недели» = Вс → сетка мастера начинается с воскресенья, логика якоря
   не меняется (проверить тестом №1 вручную).
6. Созданный «учебный» будильник открывается в обычном редакторе цикла и правится
   без потерь.
7. **Заочник**: рабочий 2/2 + период «Сессия» на месяц → рабочий молчит весь
   период; учебный будильник, включённый на этот месяц, звонит по своей сетке;
   после окончания периода рабочий возвращается — проверить оба положения
   «Заморозить цикл» (фаза продолжилась / фаза шла по календарю).

## 6. Ограничения

- Не менять: движок `schedule/` (только добавление чистого билдера рядом),
  Room-схему и миграции, формат `cycleSpec`, `AlarmScheduler`/ресиверы,
  существующие редакторы (кроме точки входа).
- Новых зависимостей не добавлять.
- Каждый этап — отдельный коммит; `testDebugUnitTest` и `assembleDebug` зелёные.
- Сборка (Windows): `JAVA_HOME="C:\Program Files\Android\Android Studio\jbr"
  GRADLE_USER_HOME="H:\gradlecache" ./gradlew.bat testDebugUnitTest assembleDebug --console=plain`.
