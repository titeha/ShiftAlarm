# Проверка настроек редактора перед сохранением

## Назначение документа

Этот документ фиксирует правила, по которым редактор ShiftAlarm разрешает или запрещает сохранение будильника.

Проверка вынесена в чистый `AlarmEditorValidator`. Она не зависит от Compose и Android API и покрывается обычными unit-тестами.

## Зачем нужен единый валидатор

Отдельные элементы интерфейса стараются не позволять собрать некорректные настройки, но UI не должен быть единственной защитой.

Например, пользовательский цикл может стать слишком длинным через комбинацию увеличения и дублирования блоков, а повреждённый `cycleSpec` может прийти из старой версии БД.

Поэтому перед сохранением весь черновик проверяется ещё раз.

## Разовый будильник

Разовый будильник не требует выбранных дней недели.

Внутреннее значение `daysMask = 0` для него корректно.

## Повтор по дням недели

Для режима «По дням недели» должен быть выбран хотя бы один день.

Если `daysMask = 0`, сохранение блокируется.

Это предотвращает неявное превращение режима «По дням недели» в разовый будильник после повторного открытия редактора.

## Встроенный сменный пресет

Если `cycleSpec == null`, используется встроенный сменный пресет.

Такой график считается корректным на уровне редактора. Корректность самих пресетов проверяется тестами движка.

## Пользовательский сменный цикл

Для своего цикла проверяются следующие правила:

```text
cycleSpec успешно декодируется;
цикл содержит хотя бы один день;
цикл содержит не более 60 дней после развёртки блоков;
цикл не требует двух разных звонков в одни календарные сутки.
```

## Несколько звонков

Несколько независимых будильников могут звонить в одни сутки. Это нормальный сценарий и данным валидатором не ограничивается.

Ограничение относится только к одному пользовательскому сменному циклу.

Ночная смена хранится на дне её отработки, но звонок для неё должен стоять вечером предыдущего календарного дня.

Конфликт возникает, если:

```text
в предыдущем дне цикла уже есть звонок на одно время;
следующая ночная смена требует на этот же день звонок на другое время.
```

Пример:

```text
День 1: дневная смена, звонок 07:00
День 2: ночная смена, звонок накануне 21:00
```

Такой цикл требует в день 1 два разных сигнала:

```text
07:00
21:00
```

Текущая модель одного цикла хранит только один звонок на календарный день, поэтому сохранение блокируется.

Пользователь должен:

```text
отключить один из звонков;
изменить структуру цикла;
или создать второй независимый будильник.
```

Если оба назначения используют одно и то же время, конфликт не создаётся: один физический сигнал может выполнить обе причины звонка.

## Поведение интерфейса

Если валидатор нашёл проблему:

```text
рядом с кнопками показывается понятное описание;
кнопка «Сохранить» становится недоступной;
черновик остаётся в редакторе;
после исправления настроек сохранение снова разрешается.
```

## Что проверяется тестами

Unit-тесты проверяют:

```text
разовый будильник;
повтор без выбранных дней;
повтор с выбранным днём;
встроенный сменный пресет;
повреждённый и пустой cycleSpec;
цикл на 60 и 61 день;
обычный блок последовательных ночей;
конфликт дневного и ночного звонков;
переход конфликта через границу цикла;
совпадающее время;
совместимость с нормализованной цепочкой ночей из шаблона.
```

## Связь с дальнейшими работами

Валидатор является последней защитой перед сохранением.

Отдельным следующим шагом контролы редактора будут ограничивать длину цикла сразу при увеличении и дублировании блоков. Даже после этого итоговая проверка останется обязательной.
