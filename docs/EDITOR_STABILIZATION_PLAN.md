# Стабилизация редактора будильника

## Назначение документа

Этот документ фиксирует найденные недочёты редактора будильника и порядок их устранения.

Работы выполняются маленькими контролируемыми шагами. Каждый шаг должен:

```text
решать одну ограниченную проблему;
по возможности иметь unit-тесты;
не менять одновременно UI, БД и движок без необходимости;
проходить текущий набор тестов;
при необходимости проверяться на реальном устройстве;
завершаться отдельным коммитом и обновлением документации.
```

## Принятое правило о нескольких звонках

Несколько независимых будильников могут звонить в одни календарные сутки. Это нормальный сценарий.

Внутри одного сменного цикла текущая модель хранит только один звонок на календарный день. Если настройки одного цикла требуют двух разных звонков в один день, это считается конфликтом.

На текущем этапе такой конфликт нужно как минимум обнаруживать до сохранения и понятно объяснять пользователю. Расширение модели до нескольких звонков одного цикла рассматривается отдельно.

## Список работ

### P0 — риск потери или искажения пользовательских настроек

- [ ] **Частичная правка периодов и диапазонов календаря.**  
  При изменении одного дня внутри длинного периода или правки должны сохраняться дни до и после выбранного диапазона. Если вырезана середина, запись разбивается на две.

### P1 — противоречивое или неожиданное поведение редактора

- [ ] **Единый валидатор редактора.**  
  Проверить перед сохранением:
  - в режиме «По дням недели» выбран хотя бы один день;
  - пользовательский цикл не пуст;
  - длина пользовательского цикла не превышает лимит;
  - `cycleSpec` корректно декодируется;
  - внутри одного цикла нет конфликта двух звонков в один календарный день.

- [ ] **Семантика режима «По выходным» для сменного графика.**  
  Привести к одному правилу планировщик, строку следующего звонка, мини-превью и календарь. До принятия правила не показывать пользователю противоречивые варианты.

- [ ] **Сохранение состояния редактора.**  
  Черновик не должен теряться при повороте экрана, пересоздании Activity или случайном нажатии «Назад».

- [ ] **Предупреждение о несохранённых изменениях.**  
  При выходе из изменённого редактора нужно предложить продолжить редактирование или отбросить изменения.

- [ ] **Подтверждение успешного сохранения.**  
  Экран закрывается только после успешной транзакции и перепланирования. При ошибке редактор остаётся открыт и показывает сообщение.

### P2 — согласованность и удобство интерфейса

- [ ] **Общий лимит длины пользовательского цикла.**  
  Лимит должен проверяться при добавлении, увеличении и дублировании блока, а также повторно перед сохранением.

- [ ] **Отмена добавления блока.**  
  Новый блок добавляется в цикл только после подтверждения в редакторе блока. Кнопка отмены не должна оставлять автоматически созданную запись.

- [ ] **Актуальное время в предпросмотре.**  
  Список и редактор должны пересчитывать ближайшие звонки по общему минутному таймеру, а не хранить `LocalDateTime.now()` до следующей композиции.

- [ ] **Разделение большого редактора на состояние и секции.**  
  Выделить `AlarmEditorUiState`, валидатор, ViewModel и небольшие composable-секции без изменения доменной логики.

- [ ] **UI/state-тесты.**  
  Основные правила редактора вынести в чистые функции и покрыть обычными unit-тестами; критичные пользовательские переходы дополнить Compose-тестами.

## Текущий шаг

### Частичная правка диапазонов

Проблема:

```text
Было: отпуск 1–14 августа.
Пользователь вернул по графику только 5 августа.
Старое поведение: удалялся весь отпуск 1–14 августа.
```

Ожидаемое поведение:

```text
Отпуск 1–4 августа.
5 августа — по графику.
Отпуск 6–14 августа.
```

То же правило действует для:

```text
периодов без будильника;
ручных правок смен;
замены периода другим типом;
замены правки другой сменой;
команды «Вернуть по графику».
```

Критерии готовности шага:

- вычитание диапазона покрыто unit-тестами;
- поддержаны отсутствие пересечения, полное удаление, обрезание начала/конца и разрезание середины;
- изменённые фрагменты получают новые Room id;
- календарный редактор использует одну чистую операцию;
- все unit-тесты проекта проходят;
- на устройстве проверен хотя бы один сценарий разрезания длинного периода и одной правки смены.

После успешной проверки этот пункт нужно отметить как выполненный:

```text
- [x] Частичная правка периодов и диапазонов календаря.
```
