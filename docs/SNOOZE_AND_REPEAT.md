# Механики пробуждения: снуз, авто-перезвон, дублирование, жёсткий режим

Документ для Claude Code. Добавляет слой «сессии звонка» поверх существующего пути
Scheduler → Receiver → Service → Activity. Расписание и движок не меняются.

Перед началом прочитай: `docs/RELIABILITY_AUDIT.md` (P1-4, P1-5), `docs/DIRECT_BOOT_FIX.md`
(RingCache, грейс-окно) — новые звонки обязаны соблюдать те же гарантии.

---

## 0. Модель: сессия звонка

Сессия = от первого срабатывания будильника до «Стоп» или «Пропущен».
Привязана к паре `(alarmId, плановое время исходного срабатывания)`.

```
ЗВОНИТ ──«Стоп»───────────────────────────► ВЫКЛЮЧЕН (сессия закрыта)
  │
  ├─«Отложить»────► СНУЗ(n+1) ──интервал──► ЗВОНИТ
  │
  └─тишина T мин──► n < M ? АВТОСНУЗ(n+1) ──интервал──► ЗВОНИТ
                    иначе  ► ПРОПУЩЕН (журнал + карточка «не был выключен»)
```

Правила:

- **Единый счётчик** `n` на сессию: ручной снуз и авто-перезвон делят общий лимит `M`.
  После «Стоп» счётчик обнуляется. Бесконечного звона не бывает.
- Дефолты (глобальные настройки приложения, новый раздел «Звонок»):
  длительность звонка `T` = 5 мин (если в коде уже есть константа — использовать её),
  интервал снуза = 5 мин (выбор 1–30), лимит `M` = 3 (выбор 0–10, 0 = снуз выключен),
  авто-перезвон = вкл (выключаемый тумблер).
- Per-alarm переопределения — **вне рамок** (редактор ждёт отдельного рефакторинга,
  `AlarmEditorScreen` в этом ТЗ не трогать).

### Хранение и надёжность

- Состояние сессии (`alarmId`, плановое время, `n`, дедлайн текущего этапа) — в
  DataStore/prefs. **Room не трогать, миграций не заводить.**
- Снуз/перезвон-звонок выставляется через существующий `AlarmScheduler`-путь
  (`setAlarmClock`), отдельный requestCode-неймспейс: `f(alarmId, "snooze")` — не
  пересекается с базовым расписанием. Extras самодостаточны (без чтения Room).
- **Снуз-звонок пишется в `RingCache`** (device-protected) с пометкой «отложен»:
  перезагрузка внутри снуз-окна не теряет звонок; грейс-окно применяется и к нему.
- Следующий базовый звонок по графику выставляется как раньше — при срабатывании
  (P1-5); снуз живёт параллельно. Теоретический конфликт «снуз позже следующего
  базового» лимитами 3×5 недостижим — зафиксировать комментарием как известное.
- Отмена сессии (снять снуз-PI, вычистить из RingCache, закрыть сессию): «Стоп»,
  выключение будильника тумблером, удаление, сохранение правок этого будильника.

### Чистая логика

Выделить `RingSessionReducer`: `(state, event) -> (state, actions)` без Android-типов.
События: `fired`, `stopPressed`, `snoozePressed`, `ringTimeout`, `cancelled`.
Действия: `scheduleSnooze(at)`, `markMissed`, `closeSession`, `journal(...)`.
Юнит-тесты: лимит исчерпан; ручной+авто вперемешку; стоп в снуз-окне; отмена;
`M = 0`; таймаут на последней попытке.

## Этап 1. Снуз

- Экран звонка: крупная кнопка «Отложить на N мин» (+ счётчик «осталось 2 из 3»),
  «Стоп» остаётся как есть. Кнопка скрывается, когда лимит исчерпан.
- Action «Отложить» в уведомлении звонка.
- Работает на заблокированном экране (путь уже directBootAware — проверить, что
  новые PendingIntent-действия тоже собираются без Room).
- Журнал: `snooze(n, until)`.
- Раздел настроек «Звонок» с дефолтами из §0.

## Этап 2. Авто-перезвон невыключенного

- Таймаут `T` без взаимодействия → авто-снуз по общей логике редьюсера,
  журнал `auto_snooze(n)`.
- Исчерпание лимита → `missed`: запись в журнал план/факт (P1-4) + ненавязчивая
  нотификация/карточка в приложении «Будильник 06:00 не был выключен» со ссылкой
  на проверку готовности.
- Сервис между перезвонами не держать: остановить звук, снять foreground,
  проснуться по снуз-звонку штатно.

## Этап 3. Дублирование будильника

- Пункт контекстного меню в списке: для режимов «разово» и «по дням недели» —
  «Дублировать +5 мин» (копия сущности, время +N, имя с суффиксом « +5»);
  для сменных — просто «Дублировать» (точная копия, создаётся **выключенной**,
  чтобы не задвоить звонки молча).
- Никакой отдельной механики «каскадов» не строить — серия покрывается снузом.

## Этап 4. Жёсткий режим (за флагом)

- Флаг компиляции `FeatureFlags.HARD_MODE` (сейчас `true`; код изолировать так,
  чтобы позже увести под платный пакет одной константой).
- Глобальная настройка «Выключение звонка»: обычное / решить пример / пройти шаги /
  встряхнуть. Применяется к кнопке «Стоп»; **«Отложить» работает всегда** —
  задание усложняет выключение, а не запирает человека в звонке.
- Задания:
  - пример: 3 подряд, уровень «двузначное ± двузначное»;
  - шаги: 20 шагов через `TYPE_STEP_DETECTOR`; датчика нет → опция скрыта в настройках;
  - встряхивание: акселерометр, ~15 встряхиваний с порогом.
- Во время задания звук продолжает играть (допустимо приглушить — на усмотрение).
- Не решил и вышел таймаут → обычная логика авто-перезвона.
- Журнал: `hard_dismiss(type, success)`.

## Ограничения

- Не менять: `schedule/` (ShiftEngine, AlarmTimes), Room-схему и миграции,
  `cycleSpec`, `AlarmEditorScreen` и редактор цикла.
- Новых зависимостей не добавлять.
- Каждый этап — отдельный коммит; `testDebugUnitTest` и `assembleDebug` зелёные.
- Сборка (Windows): `JAVA_HOME="C:\Program Files\Android\Android Studio\jbr"
  GRADLE_USER_HOME="H:\gradlecache" ./gradlew.bat testDebugUnitTest assembleDebug --console=plain`.

## Ручная проверка (человек, на устройстве)

1. Снуз с заблокированного экрана; звонок возвращается через интервал.
2. Лимит: 3 снуза → четвёртого предложения нет; таймаут после лимита → карточка «не был выключен».
3. Телефон в тишине, не трогать: авто-перезвон M раз → missed-карточка.
4. Перезагрузка **внутри снуз-окна**, не разблокировать → снуз-звонок звенит (RingCache + грейс).
5. Выключение будильника тумблером во время снуз-окна → тишина, сессия закрыта.
6. «Дублировать +5» у разового; у сменного — копия создана выключенной.
7. Жёсткий режим «шаги»: «Стоп» требует прогулки, «Отложить» работает сразу.
8. Смена системного времени во время снуза → поведение зафиксировать в отчёте.
