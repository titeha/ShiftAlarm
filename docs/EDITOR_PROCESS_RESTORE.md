# Восстановление редактора после уничтожения процесса

## Назначение документа

Этот документ фиксирует второй этап сохранения состояния редактора:

```text
восстановление подтверждённого черновика после системного уничтожения процесса.
```

Поворот и другие конфигурационные изменения уже закрываются
`AlarmEditorSession` в `AlarmListViewModel`.

## Когда Android восстанавливает состояние

Для редактора используется `SavedStateHandle`.

Состояние может быть восстановлено, если:

```text
пользователь оставил приложение в task stack;
Android уничтожил фоновый процесс для освобождения памяти;
пользователь вернулся в сохранённую задачу.
```

Состояние намеренно не восстанавливается после:

```text
Force stop;
смахивания задачи из списка недавних;
явного закрытия редактора;
перезагрузки устройства, если task stack исчез;
успешного сохранения.
```

Это соответствует назначению Android saved state.

## Что сохраняется

Компактный снимок содержит:

```text
исходный AlarmEntity;
исходные periods и overrides;
текущий сырой черновик AlarmEntity;
текущие periods и overrides;
выбранный EditMethod;
флаг открытого dirty-диалога.
```

Исходные и текущие значения нужны одновременно, чтобы после восстановления
корректно продолжила работать проверка несохранённых изменений.

## Почему сохраняется сырой черновик

В редакторе временно живут настройки нескольких способов расписания.

Например, пользователь может:

```text
настроить дни недели;
перейти в сменный режим;
потом вернуться к дням недели.
```

Поэтому снимок хранит `draftState`, а не только результат `effective(...)`.

## Формат

Снимок кодируется собственным версионированным бинарным форматом:

```text
версия;
AlarmEntity;
списки periods/overrides;
текущий метод;
UI-флаг;
URL-safe Base64.
```

Кодек не зависит от Android API и покрывается unit-тестами.

Повреждённый или неизвестный формат игнорируется без падения приложения.

## Ограничение размера

Saved state предназначен для небольшого пользовательского состояния.

Снимок ограничен примерно:

```text
96 КиБ до Base64;
128 КиБ после Base64;
до 1000 periods;
до 1000 overrides.
```

Если снимок превышает лимит:

```text
текущая ViewModel-сессия продолжает работать;
поворот экрана не теряет черновик;
process-death восстановление для этой сессии отключается;
в Logcat записывается предупреждение.
```

Произвольно большие данные должны храниться в Room, а не в Bundle.

## Когда снимок обновляется

`AlarmListViewModel` наблюдает Compose Snapshot State через `snapshotFlow`.

При каждом подтверждённом изменении обновляются:

```text
название;
время;
режим;
cycleSpec;
праздничные настройки;
periods;
overrides;
dirty-диалог.
```

При закрытии сессии ключ удаляется из `SavedStateHandle`.

## Состояние Saving

Состояние активной транзакции не восстанавливается как `Saving`.

После уничтожения процесса ViewModel создаётся заново в `Idle`.

Room-транзакция либо завершилась, либо была отменена. Список затем получает
актуальные данные из Room Flow. Черновик восстанавливается только как
редактируемое состояние.

## Проверка тестами

Unit-тесты проверяют:

```text
полный round-trip;
Unicode и специальные символы;
повреждённый Base64;
ограничение размера;
восстановление изменённой сессии;
сохранение dirty-state;
чистую сессию без ложных изменений.
```

## Проверка на устройстве

Для настоящего process death:

```text
открыть и изменить редактор;
нажать Home;
выполнить adb shell am kill ru.titeha.shiftalarm;
вернуться в задачу из экрана недавних приложений.
```

Не использовать:

```text
adb shell am force-stop;
смахивание задачи из недавних.
```

Они удаляют task state, и восстановление не ожидается.

## Правило для будущих изменений

Новое поле основного черновика должно входить в snapshot.

Крупное или долговременное состояние следует хранить в Room, а в
`SavedStateHandle` оставлять только компактный идентификатор или небольшой draft.
