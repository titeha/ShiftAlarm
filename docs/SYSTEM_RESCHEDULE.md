# Перепланирование будильников после системных событий

## Назначение документа

Этот документ фиксирует правило перепланирования будильников после системных событий Android.

Для приложения-будильника недостаточно один раз поставить сигнал в `AlarmManager`. Системное расписание нужно восстанавливать после событий, которые могут сбросить или сделать неактуальным ранее запланированное срабатывание.

## Почему это нужно

`AlarmManager` не является постоянным хранилищем расписания.

Источник истины в ShiftAlarm — база данных приложения:

```text
alarms
alarm_periods
alarm_overrides
```

Системный AlarmManager содержит только ближайшее техническое срабатывание. Его можно потерять или сделать неактуальным при системных событиях.

## Обрабатываемые события

BootReceiver обрабатывает следующие события:

```text
android.intent.action.BOOT_COMPLETED
android.intent.action.MY_PACKAGE_REPLACED
android.intent.action.TIME_SET
android.intent.action.TIMEZONE_CHANGED
```

## BOOT_COMPLETED

Событие приходит после загрузки устройства.

После перезагрузки Android не сохраняет ранее поставленные alarms как постоянное расписание приложения. Поэтому включённые будильники нужно заново прочитать из базы и поставить ближайшие срабатывания.

## MY_PACKAGE_REPLACED

Событие приходит после установки новой версии приложения поверх старой.

После обновления приложения безопаснее заново перепланировать все включённые будильники, потому что код расчёта расписания, формат данных или правила планирования могли измениться.

## TIME_SET

Событие приходит после ручного изменения системного времени.

Если пользователь перевёл время вперёд или назад, ранее рассчитанное ближайшее срабатывание может стать неверным. Поэтому включённые будильники пересчитываются заново.

## TIMEZONE_CHANGED

Событие приходит после изменения часового пояса.

ShiftAlarm рассчитывает расписание в локальных датах и локальном времени. При смене часового пояса ближайшее срабатывание нужно пересчитать относительно нового локального времени.

## Порядок обработки

При получении поддерживаемого события receiver выполняет:

```text
прочитать включённые будильники из базы;
для каждого будильника загрузить нужные периоды и правки календаря;
пересчитать ближайшее срабатывание;
поставить новый AlarmManager.setAlarmClock или отменить старый сигнал, если звонить больше нечему.
```

## Что не делает receiver

Receiver не создаёт новые будильники.

Receiver не меняет настройки пользователя.

Receiver не исправляет данные в базе.

Receiver только синхронизирует системный AlarmManager с текущим состоянием базы.

## Проверки тестами

Чистая логика выбора системных событий вынесена в SystemRescheduleActions.

Unit-тесты проверяют:

```text
BOOT_COMPLETED требует перепланирования;
MY_PACKAGE_REPLACED требует перепланирования;
TIME_SET требует перепланирования;
TIMEZONE_CHANGED требует перепланирования;
null и неизвестные действия игнорируются;
для логов возвращается читаемая причина события.
```

## Проверка на устройстве

Полноценная проверка BOOT_COMPLETED требует реальной перезагрузки устройства.

Минимальный ручной сценарий:

```text
создать включённый будильник на ближайшее будущее;
перезагрузить телефон;
после разблокировки открыть приложение;
убедиться, что ближайший будильник снова отображается системной иконкой;
дождаться срабатывания.
```

Проверку TIME_SET и TIMEZONE_CHANGED нужно выполнять осторожно, потому что изменение системного времени и часового пояса влияет на другие приложения.

## Правило для будущих изменений

Если в будущем появятся новые события, влияющие на расписание, они должны добавляться в SystemRescheduleActions, manifest и unit-тесты одним маленьким изменением.
