# Продуктовое видение ShiftAlarm

## Назначение документа

Этот документ фиксирует продуктовую цель ShiftAlarm и границы проекта.

Он нужен, чтобы дальнейшие технические решения, UI, праздничные модули, монетизация и подготовка к публикации в магазине развивались в одну сторону, а не превращали приложение в очередной «умный будильник ради умности».

## Цель проекта

ShiftAlarm должен стать полноценным Android-приложением будильника для людей, у которых расписание не укладывается в простую пятидневку.

Основная цель:

```text
надёжно будить пользователя тогда, когда ему действительно нужно вставать,
и не будить тогда, когда пользователь отдыхает, болеет, в отпуске или не учится.
```

Проект должен быть доведён до качества, с которым его можно публиковать в магазине без стыда перед пользователями.

Критерий качества:

```text
пользователь доверяет приложению свой сон и рабочее/учебное расписание;
приложение не требует ежедневного ручного включения и отключения будильника;
приложение объясняет свои решения и не ведёт себя неожиданно.
```

## Для кого приложение

### Работники сменных графиков

Главная аудитория — люди, работающие не по графику 5/2:

```text
2/2;
сутки/трое;
утро/день/ночь;
вахты;
индивидуальные циклы;
плавающие смены;
нестандартные рабочие недели.
```

Для них обычный будильник неудобен: его приходится постоянно включать, выключать, переносить и проверять вручную.

ShiftAlarm должен убрать эту ежедневную ручную возню.

### Работники с отпусками, больничными и отгулами

Отдельная важная задача — периоды без будильника.

Пользователь уходит в отпуск, на больничный, отгул или период «за свой счёт» — и приложение не должно его будить.

Правило:

```text
начался период отдыха — будильник молчит;
период закончился — будильник возвращается к рабочему расписанию.
```

Пользователь не должен помнить, что нужно вручную выключить будильник в начале отпуска и включить его обратно после отпуска.

### Студенты и школьники

Вторая перспективная аудитория — учащиеся.

У студентов и школьников тоже часто есть расписание, которое плохо ложится на простой недельный будильник:

```text
учёба шесть дней в неделю;
плавающие пары;
заочное обучение;
сессии;
каникулы;
дни без занятий;
разные расписания по неделям.
```

Для них ShiftAlarm может стать не «рабочим будильником», а учебным будильником.

Продуктовая идея та же:

```text
не надо каждый вечер думать, включать ли будильник на завтра;
приложение само знает расписание и периоды без занятий.
```

## Чем ShiftAlarm не является

ShiftAlarm не должен превращаться в «умный будильник» с искусственными препятствиями для отключения.

Не является целью:

```text
заставлять пользователя решать задачи;
трясти телефон;
сканировать QR-коды;
делать фотографии;
выполнять ритуалы для отключения.
```

Такие механики могут быть полезны отдельным людям, но это другой продукт.

Для ShiftAlarm главная «умность» — не в шаманском отключении, а в правильном расписании.

Формула продукта:

```text
не хитро выключить будильник,
а правильно понять, когда он вообще должен звонить.
```

## Международность

Проект не должен быть завязан только на Россию.

Сменная работа, ночные смены, отпуска, больничные и нестандартная учёба есть в разных странах.

Поэтому базовая архитектура должна быть международной:

```text
движок расписаний не зависит от конкретной страны;
праздники подключаются как региональные модули;
правила отпусков и переносов могут отличаться по странам;
локализация должна быть предусмотрена заранее.
```

## Праздники и производственные календари

Поддержка праздников уже заложена в движке, но полноценная загрузка, обновление, парсинг и планирование по внешним календарям ещё не доведены до реализации.

Целевая модель:

```text
ядро приложения умеет учитывать статус дня;
региональный модуль поставляет данные по стране/региону/году;
пользователь выбирает нужную страну или календарь;
будильник учитывает праздники, переносы, рабочие субботы и локальные правила.
```

Для России правила особенно сложные:

```text
официальные праздники;
переносы выходных;
рабочие субботы;
сокращённые дни;
производственный календарь;
региональные особенности в перспективе.
```

Но архитектура не должна предполагать, что Россия — единственный случай.

Первый практический порядок стран можно держать таким:

```text
Россия;
Казахстан;
другие страны по мере появления источников данных и реального спроса.
```

## Монетизация

Монетизация желательна, но не должна ломать доверие к будильнику.

Недопустимая модель:

```text
базовый будильник ненадёжен без оплаты;
критичные функции срабатывания спрятаны за paywall;
реклама мешает постановке или отключению будильника;
пользователь не понимает, за что платит.
```

Допустимые направления монетизации:

```text
платные региональные праздничные модули;
платные расширенные профили календарей;
премиум-настройки внешнего вида;
пакет расширенной диагностики и резервирования;
поддержка нескольких учебных/рабочих профилей;
пожизненная покупка Pro;
подписка только там, где есть регулярная ценность: обновляемые календари, страны, источники данных.
```

Базовый принцип:

```text
надёжный будильник и сохранность пользовательских данных должны оставаться честными;
монетизация продаёт удобство, расширение и региональные данные, а не исправление искусственных ограничений.
```

Если приложение распространяется через Google Play и продаёт цифровые функции внутри приложения, монетизацию нужно проектировать с учётом правил Google Play Billing.

## Качество перед публикацией

Перед публикацией в магазине проект должен пройти отдельный этап стабилизации.

Минимальные направления:

```text
автоматический CI на каждый push и pull request;
расширенные unit-тесты движка расписаний;
инструментальные тесты Room и миграций;
ручная матрица проверок на реальном телефоне;
диагностика разрешений и ближайшего срабатывания;
политика backup/restore;
privacy policy;
release checklist;
чёткое описание функций в магазине;
отдельный план монетизации.
```

Для приложения-будильника качество проверяется не количеством экранов, а предсказуемостью:

```text
звонит, когда должен;
молчит, когда должен;
после перезагрузки не забывает расписание;
после обновления не теряет настройки;
после отпуска возвращается к правильной смене;
ночные смены будят накануне;
праздники и переносы применяются объяснимо.
```

## Позиционирование

Рабочее позиционирование:

```text
ShiftAlarm — будильник для сменной работы, нестандартной учёбы и реального календаря жизни.
```

Более бытовая формулировка:

```text
будильник, который понимает твой график, отпуск, ночные смены и праздники.
```

Что важно подчеркнуть пользователю:

```text
это не будильник с трюками для отключения;
это будильник, который заранее знает, когда тебе вставать;
он подходит для сменщиков, вахтовиков, студентов и школьников;
он не будит в отпуске, больничном и на выходных, если так настроено;
он может учитывать праздники и переносы.
```

## Связь с технической дорожной картой

Этот документ отвечает на вопрос «зачем делаем».

Техническая дорожная карта отвечает на вопрос «что привести в порядок».

Приоритет дальнейших работ:

```text
сначала надёжность ядра и Android-интеграции;
потом CI, диагностика, разрешения и релизная дисциплина;
потом праздники и международные календарные модули;
потом UX/UI-полировка;
потом монетизация и публикация.
```

## Правило для будущих решений

Если новая функция не помогает хотя бы одной из этих задач:

```text
меньше ручной возни с будильником;
точнее учитывать реальный график;
надёжнее срабатывать;
понятнее объяснять пользователю расписание;
лучше учитывать отпуск, праздники, учёбу или смены;
```

её нужно отложить.

ShiftAlarm должен оставаться прикладным инструментом, а не коллекцией эффектных, но лишних возможностей.
