# Обработка ошибок операций вне редактора

## Назначение

Операции над будильниками из списка — включение/выключение, удаление, открытие
редактора — раньше запускались как fire-and-forget корутины без обработки ошибок.
Исключение репозитория Room или планировщика `AlarmManager` тихо терялось (или
роняло корутину), и пользователь не получал никакого сигнала.

Этот шаг вводит **единое правило**: техническая деталь — в журнал, пользователю —
короткое понятное сообщение через Snackbar.

## Механизм

`AlarmListViewModel` отдаёт одноразовые сообщения:

```text
private val _userMessages = Channel<String>(Channel.BUFFERED)
val userMessages: Flow<String> = _userMessages.receiveAsFlow()
```

Канал (а не `StateFlow`) выбран специально: это **события**, а не состояние. Сообщение
показывается один раз и не переигрывается при пересоздании Activity.

Операции обёрнуты по одному правилу:

```text
try {
    // работа с репозиторием и планировщиком
} catch (error: CancellationException) {
    throw error                      // отмену корутины не глушим
} catch (error: Exception) {
    Log.e(TAG, "...", error)         // техническое — в Logcat
    _userMessages.send("...")        // пользователю — короткое сообщение
}
```

Важно: `CancellationException` **перебрасывается**. `openEditor` отменяет предыдущую
загрузку (`openEditorJob?.cancel()`), и без реброса штатная отмена показала бы ложную
ошибку.

Сообщения:

```text
setEnabled → «Не удалось изменить будильник»
delete     → «Не удалось удалить будильник»
openEditor → «Не удалось открыть будильник»
```

## Показ на экране

`MainActivity` собирает поток и показывает Snackbar поверх любого экрана:

```text
val snackbarHostState = remember { SnackbarHostState() }

LaunchedEffect(Unit) {
    vm.userMessages.collect { message ->
        snackbarHostState.showSnackbar(message)
    }
}
```

`SnackbarHost` добавлен оверлеем в общий `Box` (без вложенного `Scaffold`), выровнен по
низу. Ошибка `openEditor` оставляет пользователя на списке — там Snackbar и виден.

## Границы шага

```text
не меняется Room и миграции;
не меняется движок расписаний;
сохранение будильника имеет отдельные состояния (AlarmSaveState) и здесь не трогается;
предупреждение AlarmManager после сохранения по-прежнему показывается своим диалогом.
```
