# Нестандартная рабочая неделя и типизированный календарь праздников

Документ для Claude Code. Цель: weekly-будильники с любым набором рабочих дней
(3, 4, 6 дней, любые дни недели) должны корректно согласовываться с государственными
праздниками и переносами; начало недели (Пн/Вс/Сб) — настраиваемое отображение.

Перед началом прочитай: `docs/HOLIDAY_CALENDAR_SOURCE.md`, код `ProductionCalendar`,
`AlarmPolarity`/`HolidayAlarms`, weekly-ветку `AlarmTimes`. Имена ниже — предположения,
адаптируйся к фактическим, намерение сохраняй.

---

## 0. Суть проблемы и принцип решения

Сейчас «выходной» — склейка двух независимых понятий:

- **личный выходной** — день недели, который не входит в рабочую неделю пользователя
  (свойство его графика);
- **государственный особый день** — праздник, перенос, сокращённый день
  (свойство страны, не зависит от пользователя).

isDayOff отдаёт булево «выходной» *для стандартной пятидневки* — для нестандартной
недели эта склейка неверна. Решение: развести оси и композировать явной матрицей.

Три новых чистых сущности в `schedule/` (без Android и I/O):

```
enum StateDayKind { NONE, HOLIDAY, TRANSFER_OFF, TRANSFER_WORK, PRE_HOLIDAY, UNKNOWN }

class WorkWeek(val workingDays: Set<DayOfWeek>)          // личная неделя
class CountryProfile(                                     // пока только RU
    val standardWeek: WorkWeek,                           // RU: Пн–Пт
    val defaultWeekStart: DayOfWeek                       // RU: Пн
)

class PersonalDayResolver(profile, typedCalendar, holidaysEnabled, polarity)
    fun ringOn(date, workWeek): Decision   // чистая функция, см. матрицу §2
```

Для weekly-будильника `workWeek` = выбранные чекбоксы дней — **новых полей и UI в
редакторе не заводить**. Ключевое правило: `transfersApplicable =
(workWeek == profile.standardWeek)` — переносы имеют смысл только для стандартной
недели, для «своей» недели действуют лишь чистые праздники.

## 1. Типизированный государственный календарь

- Заменить булев статус дня на `StateDayKind` (см. enum выше).
- **Встроенные данные RU** (авторитетный источник для типов): расширить существующий
  оффлайн-fallback RF-2026 до типизированного списка особых дат
  `{date, kind}` — праздники, переносы обоих направлений, сокращённые дни.
  Добавить RF-2027, если официальные данные уже опубликованы; иначе — оставить
  комментарий-заглушку.
- **isDayOff остаётся** источником обновления для стандартной части: сверься с
  `docs/HOLIDAY_CALENDAR_SOURCE.md` и документацией API — какие параметры реально
  поддерживаются (сокращённые дни, шестидневка). **Параметры API не выдумывать**;
  чего нет в документации — того нет. Типы, которые из API не выводятся однозначно
  (праздник на субботе vs просто суббота), берутся из встроенного списка.
- Формат локального кэша: поднять версию; кэш пересоздаваемый — **никаких
  Room-миграций**. Если кэш сейчас в Room — зафиксируй в отчёте и предложи вынос
  в файл/DataStore отдельным решением, не делай молча.
- `UNKNOWN` (нет данных на дату/страну): вести себя как `NONE` + пометка в журнале.

## 2. Матрица композиции (сердце документа)

Полярность «по рабочим» (глушить, когда не работаю). `P` = день недели входит в
личную неделю; `T` = transfersApplicable.

| StateDayKind    | Звонок?                                        |
|-----------------|------------------------------------------------|
| NONE / UNKNOWN  | звонит ⇔ P                                     |
| HOLIDAY         | **тишина всегда** (праздник касается всех)     |
| TRANSFER_OFF    | T → тишина; не T → звонит ⇔ P                  |
| TRANSFER_WORK   | T → **звонит** (даже если день не отмечен: стандартная неделя следует переносам); не T → звонит ⇔ P |
| PRE_HOLIDAY     | как NONE (подъём тот же); тип виден в календаре |

Полярность «по выходным» (будить, чтобы не залёживаться): вычислить «рабочий ли
день» по матрице выше и **инвертировать**: тишина в рабочие, звонок в личные
выходные, праздники и TRANSFER_OFF. Итог для шестидневки Пн–Сб: будит только в
воскресенье и праздники.

Тумблер «Учитывать праздники» выключен → вся матрица схлопывается в «звонит ⇔ P»
(текущее поведение недельного режима).

**Сменный режим (MODE_SHIFT)**: личной недели нет — цикл сам определяет рабочие дни.
Из государственного календаря на смены влияет **только HOLIDAY** (и полярность),
как сейчас в `HolidayAlarms`; TRANSFER_* к циклам не применяются. Поведение не
менять, только перевести чтение на типизированный kind + регрессионные тесты.

Приоритеты резолва не меняются: исключение > подмена > период > базовое правило.

## 3. Начало недели

- Глобальная настройка «Начало недели»: `авто` (из локали, `WeekFields.of(Locale)`) /
  Пн / Вс / Сб. По умолчанию — авто.
- Влияет **только на отображение**: порядок колонок сетки календаря, порядок
  чипов дней недели в редакторе (только порядок — структуру редактора не трогать),
  подписи строк. Движок работает с абсолютными датами, `weekStart` туда не проникает.
- Чистый помощник `orderedDaysOfWeek(start): List<DayOfWeek>` + тест.

## 4. Этапы (коммиты)

- **Этап 0 — разведка**: где живут ProductionCalendar/кэш/fallback, как weekly-ветка
  сейчас применяет праздники и что означает «переносы» в текущем коде. Короткий
  отчёт-план, без правок.
- **Этап 1 — типизированный календарь** (§1) + тесты парсинга и fallback.
- **Этап 2 — WorkWeek + PersonalDayResolver + матрица** (§0, §2), интеграция в
  weekly-путь `AlarmTimes`; сменный путь — только смена типа чтения. Golden-тесты
  на реальных датах RU (см. §5-тесты).
- **Этап 3 — календарь UI**: в деталях дня показывать причину («праздник»,
  «перенос: рабочая суббота», «сокращённый день»); покраску не перестраивать.
- **Этап 4 — начало недели** (§3): настройка + сетка + порядок чипов.

## 5. Тесты

Юнит (обязательные, чистые, в `app/src/test/.../schedule/`):

1. Пятидневка Пн–Пт, TRANSFER_WORK-суббота → звонит; TRANSFER_OFF-будни → тишина.
2. Шестидневка Пн–Сб: HOLIDAY на субботе → тишина; обычная суббота → звонит;
   TRANSFER_WORK-суббота → просто звонит (не дублируется, не ломается).
3. Четырёхдневка Пн–Чт: TRANSFER_WORK-суббота → тишина (переносы не применимы);
   HOLIDAY на вторнике → тишина.
4. Трёхдневка со «странным» набором (Вт, Чт, Сб): праздники глушат, переносы игнорируются.
5. «По выходным» для шестидневки: звонит Вс и в HOLIDAY, молчит Пн–Сб.
6. Тумблер праздников выключен → чистое «по галочкам» во всех случаях выше.
7. UNKNOWN-дата → как NONE + журнальная пометка.
8. `orderedDaysOfWeek` для Пн/Вс/Сб.

Ручная проверка: календарь показывает причины дня; смена «Начало недели» на Вс
перестраивает сетку и чипы; локаль ja-JP в режиме «авто» даёт воскресенье.

## 6. Ограничения

- Не менять: Room-схему и миграции, `cycleSpec`, структуру редакторов (только
  порядок чипов), `AlarmScheduler`/ресиверы. Новых зависимостей не добавлять.
- Официальная «шестидневка по производственному календарю» (отдельный
  государственный 6-дневный календарь) — **вне рамок**: до появления типизированного
  источника Пн–Сб трактуется как «своя неделя» (только праздники). Зафиксировать
  комментарием у CountryProfile.
- Мультистрана — вне рамок; но `CountryProfile` — точка расширения под платный
  модуль по странам из планов README.
- Каждый этап — отдельный коммит; `testDebugUnitTest` и `assembleDebug` зелёные.
- Сборка (Windows): `JAVA_HOME="C:\Program Files\Android\Android Studio\jbr"
  GRADLE_USER_HOME="H:\gradlecache" ./gradlew.bat testDebugUnitTest assembleDebug --console=plain`.
