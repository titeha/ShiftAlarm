# Снуз (сессия звонка) — этапы 1–2

Реализация по `SNOOZE_AND_REPEAT.md`. Слой «сессия звонка» поверх пути
Scheduler → Receiver → Service → Activity; движок и расписание не тронуты.

## Модель

- **Чистое ядро** `alarm/RingSessionReducer` — `reduce(state, event, config, now) → (state, actions)`,
  без Android, покрыто unit-тестами. События: fired/stopPressed/snoozePressed/ringTimeout/cancelled;
  действия: ScheduleSnooze / MarkMissed / CloseSession / Journal.
- **Единый счётчик** снуза на сессию: ручной снуз и авто-перезвон (этап 2) делят общий лимит `M`.
  «Стоп»/отмена закрывают сессию и обнуляют счётчик — бесконечного звона нет.
- Сессия привязана к `alarmId` (одна активная сессия на будильник).

## Хранение и надёжность

- `RingSessionStore` — device-protected prefs `ring_sessions` (переживает ребут в снуз-окне,
  доступно до разблокировки). Кодек версионирован, покрыт тестом.
- Снуз-звонок ставится через `AlarmScheduler.scheduleSnoozeAt` в ОТДЕЛЬНОМ (отрицательном)
  requestCode-неймспейсе (`AlarmRequestCodes.snooze`) и пишется в снуз-список RingCache
  (`DirectBootAlarmStore.putSnooze`, ключ `snoozes_v1` — базовый кэш не тронут). После ребута
  `reArmFromCache` перевыставляет и снузы (то же грейс-окно).
- Снуз-срабатывание помечено `EXTRA_IS_SNOOZE`: `AlarmReceiver` звонит по метке из extras БЕЗ
  валидатора актуальности и без Room (direct-boot-safe); «использованный» снуз убирается из кэша.

## UI и отмена

- Экран звонка (`AlarmActivity`): кнопка «Отложить на N мин · осталось K», пока лимит не исчерпан.
- Уведомление звонка: action «Отложить N мин» (пока доступно) + «Стоп».
- `RingSessionController` — исполнитель редьюсера (грузит настройки/сессию, выполняет действия);
  журнал locked-safe (CE → DPS-буфер `DirectBootEventBuffer`).
- Сессия закрывается (снуз снимается) при «Стоп», выключении тумблером, удалении, сохранении правок.
- Настройки → «Звонок»: интервал снуза (5/10/15), лимит `M` (0 = снуз выключен), тумблер
  «Авто-перезвон невыключенного» и время до него `T` (5/10/15).

## Авто-перезвон невыключенного (этап 2)

- Пока сервис звенит (жив, с wake-lock), заведён `Handler`-таймаут на `T` минут. По истечении —
  `RingEvent.RingTimeout` через редьюсер: авто-снуз (в пределах общего лимита `M`) или, если исчерпан —
  «пропущен». Между перезвонами сервис не держим: звук глохнет, foreground снимается, просыпаемся по
  снуз-звонку штатно. Таймаут снимается при «Стоп»/снузе/остановке.
- Тумблер выключен → таймаут не ставится, звонок звенит до ручного «Стоп» (прежнее поведение).
- Исчерпание лимита → ненавязчивая нотификация «Будильник не был выключен» (канал `missed_channel`,
  открывает приложение) + событие `MISSED` в журнал. Карточку надёжности («мог не прозвенеть» / OEM-
  выгрузка) НЕ трогаем — это другой случай.

## Дублирование будильника (этап 3)

Меню «⋮» в строке списка → «Дублировать». Разовый / по дням недели — копия со сдвигом времени
**+5 мин**, суффиксом « +5» в имени, включённая. Сменный — точная копия с периодами/правками, но
**выключенная** (чтобы не задвоить звонки молча; включит пользователь). Каскадов нет — серия покрыта
снузом. `AlarmListViewModel.duplicate` (сменный — через `saveWithChildren`).

## Журнал

Типы `SNOOZED` (ручной/авто) и `MISSED` (звонок не был выключен; важный). Видны в диагностике.
