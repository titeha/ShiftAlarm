# Консолидированное ревью и план стабилизации ShiftAlarm

Дата: 16 июля 2026 года.

Подготовлено на основе:

```text
текущего состояния проекта;
первичного аудита ShiftAlarm;
последовательности уже выполненных исправлений;
дополнительного ревью другого ИИ;
ручных проверок на устройстве.
```

Анализ и консолидация: ChatGPT (GPT-5.6 Pro).

## 1. Краткий вывод

Дополнительное ревью содержит полезные направления, но смешивает:

```text
реальные замечания;
уже закрытые проблемы;
предположения без подтверждения кодом;
несуществующие компоненты проекта;
слишком общие архитектурные рекомендации.
```

Его нельзя принимать как готовый план без проверки.

На текущем этапе в проекте не обнаружен новый подтверждённый P0-дефект уровня
потери данных или гарантированного несрабатывания. Основные оставшиеся задачи
относятся к устойчивости ошибочных путей, сохранению состояния редактора,
часовым поясам/DST, разрешениям и доступности интерфейса.

## 2. Проверка шага с полярностью REST

В режиме «По графику» пользователь больше не должен видеть вариант
«По выходным».

Это ожидаемое поведение, а не неполная проверка.

Для сменного графика принято правило:

```text
время звонка берётся только из сменного цикла;
производственный календарь глушит звонок в официальный нерабочий день;
REST не применяется к сменному циклу;
для отдельного звонка по выходным создаётся отдельный простой будильник.
```

Старую запись `mode=shift, polarity=REST` невозможно нормально создать через
новый UI. Её совместимость проверяется unit-тестами политики и `AlarmTimes`.

Шаг можно считать закрытым, если:

```text
в сменном режиме REST отсутствует;
превью использует времена смен;
праздничные дни глушат звонки графика;
простой режим по выходным продолжает работать;
unit-тесты проходят.
```

## 3. Расхождения с дополнительным ревью

### 3.1. Архитектура и Hilt

Утверждение:

```text
Clean Architecture с domain/UseCase;
schedule/ — отдельный модуль;
используется Hilt.
```

Статус: неверно.

Фактическая структура:

```text
alarm/
data/
schedule/
ui/
MainActivity
AlarmActivity
```

`domain/` отсутствует.

`schedule/` — чистый пакет внутри единственного Gradle-модуля `:app`,
а не отдельный модуль.

Hilt не подключён. Зависимости создаются вручную через `Application`,
ViewModel и конструкторы.

Корректная формулировка:

```text
слоистая архитектура с чистым доменным ядром расписаний;
не строгая Clean Architecture;
без DI-контейнера.
```

Добавлять Hilt только ради формального соответствия архитектурному шаблону
не требуется. DI следует вводить, когда ручная сборка зависимостей реально
мешает тестированию или развитию проекта.

### 3.2. Часовые пояса и DST

Замечание о необходимости проверить DST — обосновано.

Рекомендация «перевести всё на Instant и использовать withZoneSameInstant»
в текущем виде неверна.

Будильник вида «каждый день в 07:00» задаёт локальное календарное время.
Если хранить только Instant, после смены часового пояса он может прозвонить
не в 07:00 по местным часам.

Текущая модель по смыслу правильная:

```text
график рассчитывается в LocalDate/LocalTime/LocalDateTime;
на границе AlarmManager локальный момент преобразуется в Instant;
при TIME_SET/TIMEZONE_CHANGED сигналы перепланируются.
```

Пробел проекта — не отсутствие `Instant`, а отсутствие явно зафиксированной
политики для двух случаев:

```text
DST gap:
локального времени не существует, например 02:30 при переводе часов вперёд;

DST overlap:
локальное время встречается дважды, например 02:30 при переводе назад.
```

Нужно решить и протестировать:

```text
gap — сдвинуть на первый допустимый момент или пропустить;
overlap — использовать первое или второе вхождение;
при поездке — следовать локальному времени устройства или домашнему часовому поясу.
```

Для текущего продукта рекомендуется:

```text
следовать локальному времени устройства;
gap — сдвигать вперёд по правилам ZoneId;
overlap — выбирать первое вхождение;
один пользовательский будильник не должен звонить дважды.
```

Перед международным запуском политика обязательна. Для первой РФ-беты это
важная, но не доказанная критическая ошибка.

### 3.3. Ошибки корутин и операций ViewModel

Замечание частично актуально.

Уже закрыто:

```text
сохранение имеет состояния Idle/Saving/Saved/Failed;
ошибка Room оставляет редактор открытым;
ошибка AlarmManager после записи показывается отдельно;
AlarmReceiver и BootReceiver защищены try/catch;
ошибки срабатывания пишутся в диагностику.
```

Остаётся проверить и привести к одному правилу:

```text
открытие дочерних данных для редактора;
включение/выключение;
удаление;
фоновая инициализация ShiftAlarmApp;
массовое rescheduleAll;
ошибки потоков Room.
```

Особенно важно: ошибка одного будильника при массовом перепланировании
не должна останавливать обработку остальных.

### 3.4. Отказ в разрешениях

Утверждение, что пользователь остаётся без объяснения, устарело.

Уже есть:

```text
проверка exact alarm;
проверка уведомлений;
проверка full-screen intent;
проверка ограничений батареи;
баннер с объяснением;
кнопки перехода в системные настройки;
повторная проверка при ON_RESUME;
диагностический экран.
```

Остаётся UX-улучшение:

```text
не показывать POST_NOTIFICATIONS автоматически сразу при первом запуске;
сначала объяснить, зачем разрешение нужно будильнику;
просить разрешение в контексте создания или включения первого будильника;
после отказа оставить постоянный понятный путь в настройки.
```

Это улучшение доверия и конверсии разрешения, а не отсутствие базовой защиты.

### 3.5. CalendarViewModel и тяжёлые вычисления

Утверждение о тяжёлой `calculateMonthShifts()` в `CalendarViewModel`
не подтверждается проектом.

Такого ViewModel сейчас нет.

Календарь строит ограниченное число ячеек месяца и использует чистые функции
движка. Нет доказательств, что расчёт создаёт заметную нагрузку на main thread.

Правильное действие:

```text
сначала измерить recomposition и время расчёта;
переносить вычисления в Dispatchers.Default только при подтверждённом джиттере;
при необходимости заранее строить CalendarMonthUiState.
```

Не следует усложнять код по предположению.

### 3.6. Единый UiState

Утверждение «UiState отсутствует» неверно для списка будильников:
`AlarmListUiState` уже объединяет будильники, периоды и правки.

Отдельные потоки не являются ошибкой сами по себе. Разделять действительно
независимые состояния допустимо.

Актуальная проблема находится в редакторе:

```text
большой объём состояния хранится через remember;
черновик не переживает пересоздание Activity;
нет SavedStateHandle;
AlarmEditorScreen остаётся слишком крупным;
часть правил уже вынесена, но state holder ещё не создан.
```

Здесь нужен `AlarmEditorViewModel` или отдельный screen-level state holder.

### 3.7. Ошибки планирования

Замечание частично актуально.

Сохранение уже различает:

```text
данные не записаны;
данные записаны, AlarmManager не обновлён.
```

Receiver и системное перепланирование логируют исключения.

Остаётся:

```text
единый тип результата планирования;
обработка ошибки при setEnabled;
изоляция ошибок в rescheduleAll;
пользовательское сообщение для операций вне редактора;
диагностическая сводка массового перепланирования.
```

### 3.8. Accessibility

Замечание об accessibility обосновано, но правило
«contentDescription для всех иконок» слишком грубое.

Декоративная иконка рядом с читаемым текстом должна оставаться без отдельного
описания, иначе TalkBack дублирует информацию.

Нужно проверить интерактивные элементы:

```text
предыдущий/следующий месяц;
ячейка календаря;
блок пользовательского цикла;
кнопки с одним символом;
переключатели и связанные подписи;
диалоги выбора времени;
контраст и размер области нажатия.
```

Для календарной ячейки полезно единое описание:

```text
15 июля, ночная смена, звонок в этот день, выбран.
```

### 3.9. Анимации

Это низкий приоритет.

Material Switch уже имеет собственную анимацию. Дополнительные crossfade и
переходы не исправляют надёжность и могут ухудшить доступность.

Анимации добавляются после завершения логики и проверки UX.

### 3.10. Обратная связь сохранения и удаления

Замечание о сохранении устарело:

```text
есть индикатор;
повторное нажатие заблокировано;
ошибка показывается;
экран закрывается после результата.
```

Для удаления уже есть подтверждение.

Возможное улучшение:

```text
Snackbar «Будильник удалён»;
в перспективе — Undo.
```

Undo требует аккуратного восстановления будильника, периодов и правок,
поэтому это отдельная функция, а не косметическая правка.

### 3.11. Легенда календаря

Замечание устарело.

Календарь уже показывает:

```text
легенду типов дней;
обозначение звонка колокольчиком;
нерабочий день для недельного календаря.
```

Можно улучшать компоновку, но отсутствие легенды не является проблемой.

## 4. Консолидированный план

### P1 — до публичной беты

- [x] **Сохранение состояния редактора.**  
  Вынести состояние в `AlarmEditorViewModel`/state holder. Переживать поворот,
  пересоздание Activity и восстановление процесса через `SavedStateHandle`.
  Закрыто: `AlarmEditorSession` (state holder в ViewModel) + шаг 11 (поворот) +
  шаг 12 (process death через SavedStateHandle).

- [x] **Единая обработка ошибок пользовательских операций.**  
  Защитить открытие редактора, toggle и delete. Ошибку показывать через
  `UiMessage`/Snackbar, технические данные писать в журнал.
  Закрыто: канал `userMessages` в ViewModel → Snackbar; см. `USER_OPERATION_ERRORS.md`.

- [x] **Изоляция массового перепланирования.**
  Один повреждённый или неразрешённый будильник не останавливает остальные.
  Возвращать отчёт `success/failed` и записывать его в диагностику.

- [x] **UX разрешений.**  
  Объяснять назначение уведомлений до системного диалога. Запрашивать разрешение
  после понятного действия пользователя, а не автоматически на старте.
  Закрыто: контекстный запрос с обоснованием при создании/включении будильника,
  один раз; путь в настройки после отказа даёт баннер готовности.
  См. `NOTIFICATION_PERMISSION_UX.md`.

- [ ] **Политика часового пояса и DST.**  
  Вынести преобразование локального времени в `Instant` в чистый resolver,
  передавать `ZoneId`, добавить тесты gap/overlap и смены зоны.

- [ ] **UI/state-тесты редактора.**  
  Проверить восстановление состояния, dirty-state, save-state, валидацию
  и основные переходы способов расписания.

### P2 — до публикации в магазине

- [ ] **Accessibility-аудит.**  
  TalkBack, семантика интерактивных элементов, контраст, touch targets,
  масштабирование шрифта.

- [ ] **CI: lint и инструментальные проверки.**  
  Добавить `lintDebug`, стабильную сборку release, запуск миграционного теста
  на эмуляторе или отдельном устройстве.

- [ ] **Device matrix.**  
  Зафиксировать результаты POCO/Redmi, Samsung и чистого Android.
  Проверить Doze, reboot, permission denial, full-screen, battery saver.

- [ ] **Полнота диагностики.**  
  Ошибки старта приложения, массового reschedule и пользовательских операций
  должны оставлять понятный след.

- [ ] **Release-профиль.**  
  Подпись, versionCode/versionName, R8-решение, privacy policy, data safety,
  release checklist.

### P3 — после стабилизации

- [ ] Snackbar после удаления и возможный Undo.
- [ ] Анимации раскрытия секций и изменения состояний.
- [ ] Профилирование календаря и Compose recomposition.
- [ ] Дополнительный DI-инструмент только при реальной необходимости.
- [ ] Расширение календарей по странам и регионам.

## 5. Рекомендуемый порядок маленьких шагов

Следующий кодовый шаг:

```text
1. Сохранение состояния редактора при повороте и пересоздании Activity.
```

Затем:

```text
2. Ошибки toggle/delete/open-editor.
3. Изоляция rescheduleAll по одному будильнику.
4. Контекстный запрос уведомлений.
5. DST resolver и тесты.
6. Accessibility-аудит.
7. CI/release/device matrix.
```

Причина такого порядка:

```text
редактор сейчас является самым крупным и изменяемым UI-компонентом;
его состояние уже защищено от явного выхода, но ещё не от конфигурации;
ошибочные пути операций — следующий реальный риск;
DST важен до международного расширения, но не требует ломать доменную модель;
анимации и косметика не должны опережать устойчивость.
```

## 6. Что не добавлять в срочный backlog

Без измерений или подтверждения не следует срочно:

```text
переписывать весь проект на Instant;
добавлять Hilt;
создавать искусственный domain-модуль;
переносить календарь на Dispatchers.Default;
добавлять анимации ради формального пункта ревью;
переписывать все StateFlow в один гигантский объект.
```

Каждое такое изменение должно решать наблюдаемую проблему, а не соответствовать
абстрактному шаблону.
